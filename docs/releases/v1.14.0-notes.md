## Clarified

### Goals / Non-goals

- **Goals:** short trip/day overviews for sharing and guidebooks, separation from Remark and Note, scan-friendly guidebook text.
- **Non-goals:** long-form travel journals, reservations, expenses, venue data, itinerary timelines, checklists, photos/attachments.

### What Summary is

Reader-oriented, short overview text for sharing, printing, and scanning. Scope: **Trip and Day** primarily.

### Trip Summary / Day Summary

| Level | Verdict |
|---|---|
| **Trip Summary** | **Yes** — trip purpose, tone, one-line context for companions |
| **Day Summary** | **Yes** — daily theme, main destinations, scan before reading itinerary rows |
| **Itinerary Summary** | **Not in first cut** — `itinerary.title` already acts as the action label |

Avoid overlap among `title` / `summary` / `note` at the Itinerary level.

### Summary vs Remark vs Note

| Concept | Role |
|---|---|
| **Summary** | Short shared overview (Trip / Day) |
| **Remark** | Short inline supplement per itinerary row (`itinerary_items.note`) |
| **Note** | Long-form free text (0..N per owner) |

### Export / import (future — not finalized)

Schema v3 extension (`trip.summary`, `day.summary`) is listed as a **candidate only**.

This is **not a committed decision**. As Summary, Participant, Photo, Attachment, and other models are organized, **Export Schema v4** at implementation time remains a realistic option. Re-evaluate in the implementation phase.
