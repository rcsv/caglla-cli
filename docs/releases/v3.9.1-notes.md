# v3.9.1 — Legacy days summary migration fix

## Summary

**v3.9.1** is a compatibility patch for existing SQLite databases whose `days` table predates the `summary` column. Opening such a DB (for example via `db status --json` on an existing `caglla.db`) could fail with `table days has no column named summary` because day backfill ran before the summary migration.

Requires **v3.9.0** baseline. No trip export schema change (still **v8**).

---

## Fixed

- **`init_db` migration order** — `migrate_summaries` now runs **before** `migrate_days`, so legacy `days` tables receive `ALTER TABLE days ADD COLUMN summary TEXT` (or `description` → `summary` rename) before any day backfill `INSERT` uses `summary`.
- **`db status --json`** — succeeds on existing legacy DBs that were missing `days.summary`.

---

## Unchanged

| Item | Notes |
|---|---|
| SQLite schema target | Same as v3.9.0 after migration completes |
| `trip export` JSON schema | Still **v8** |
| `db path` / `db status` on missing DB | Still do **not** create the database file |
| DB path resolution (`--db` / `CAGLLA_DB` / `caglla.toml`) | Unchanged from v3.9.0 |

---

## Upgrade notes

If `cargo run -- db status --json` failed on an older `caglla.db` with the summary column error, upgrade to **v3.9.1** and rerun the same command. The migration applies automatically on first open.
