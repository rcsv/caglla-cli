# v4.6.1 — SQLite FK / orphan data hardening review

## Summary

**v4.6.1** is a **documentation-only review** after v4.6.0 TripStats.days semantics fix. It inventories SQLite foreign key coverage, orphan data risks, and application-level cascade delete behavior—without schema or migration changes.

Review: [v4.6.1-sqlite-fk-orphan-data-hardening-review.md](../specifications/v4.6.1-sqlite-fk-orphan-data-hardening-review.md)

## Headline findings

- **DB FK coverage is partial** — only `days`, `itinerary_items`, and `checklist_items` reference `trips` with `ON DELETE CASCADE`; `itinerary_items.day_id` → `days` is `NO ACTION`
- **Most child data has no FK** — `expenses`, `estimates`, `reservations`, `notes`, `participants`, `receipts`, `expense_beneficiaries`
- **CLI delete paths are safe** when using app commands — `trip delete` / `itinerary delete` explicitly cascade children first
- **Raw SQL or future multi-process access is high risk** — app-level delete order is not enough alone
- **Do not add FK constraints immediately** — orphan detection and migration strategy must come first

## Recommended sequence

```text
1. v4.6.1 — FK / orphan review (this release)
2. v4.6.2 — SQLite migration strategy review
3. v4.6.x — orphan detection / doctor integration
4. v4.6.x — migration runner / backup / dry-run
5. v4.7.x+ — FK introduction, table by table
```

## Non-goals

- Schema / migration / FK changes
- Orphan auto-repair
- Receipt / Expense / Travel Book behavior changes
- Code or golden updates
