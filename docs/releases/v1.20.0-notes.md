## Review conclusion

```text
Summary is not a user travel journal.
Summary is an Abstract layer — a trip-level overview, primarily for system/AI generation,
serving as input to Travel Book generation.
```

## Added

- `docs/specifications/summary-post-implementation-review.md` — validates v1.17.0 and redefines Summary as Abstract.

## Reviewed (documented)

### Responsibility model

```text
Note            = Fact   (user input — memos, facts, fragments)
Summary         = Abstract (generated overview — not a journal)
Travel Journal  = Story  (future — experiences, reflections, timeline)
```

### Why the change from v1.14

v1.14 treated Summary as a **user-written short overview**. After v1.17 implementation and Travel Journal / Travel Book roadmap review, user-journal use of Summary would **conflict with Travel Journal**. Responsibilities are now separated.

**v1.14 document is not overwritten** — it remains valid design history for its time.

### v1.17 implementation alignment

| Layer | Verdict |
|---|---|
| **DB** (`trips.summary`, `days.summary`) | **Adequate** — author-neutral text storage; no schema change needed |
| **CLI** (`trip add/update --summary`, `day update --summary`) | **Valid** — interim **manual bootstrap / override**; not deprecated in v1.20 |
| **Export / import v3** | **Maintained** — human or generated text both OK |
| **export-md** | **Maintained** — Travel Book input material |
| **trip diff** | **Maintained** — Summary changes after Trip Data / regeneration are natural |

### Travel Book pipeline

```text
Trip Data → Summary Generator → Summary → Travel Book Generator → Travel Book
```
