# v4.6.2 — SQLite migration strategy review

## Summary

**v4.6.2** is a **documentation-only review** after v4.6.1 FK / orphan data hardening. It defines how SQLite schema should evolve safely before introducing FK constraints or a formal migration runner.

Review: [v4.6.2-sqlite-migration-strategy-review.md](../specifications/v4.6.2-sqlite-migration-strategy-review.md)

## Headline findings

- **Current model:** implicit **migrate-on-open** via `init_db` on every `open_db_at` — idempotent `migrate_*` functions, no `PRAGMA user_version`
- **`db status` `schema_version`** is a CLI JSON constant — **not** the SQLite DB migration version (distinct from trip export `schema_version` v8)
- **Table rebuild precedent exists** — `migrate_receipts_simplified_schema`
- **FK introduction requires migration strategy first** — orphan detection, backup, dry-run before destructive steps

## Recommended strategy

```text
Short term:  PRAGMA user_version for DB schema version; explicit migration steps
Medium term: backup / dry-run / report-only orphan checks
Long term:   migration runner; then table-by-table FK (v4.6.1 order)
```

## Non-goals

- Schema / migration runner / FK / orphan detection implementation
- Rust functional changes
- trip export/import schema changes
