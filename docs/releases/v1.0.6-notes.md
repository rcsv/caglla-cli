# v1.0.6 - Structured doctor/advisor JSON

## Summary

- **`trip doctor --json`**: Root output is now an envelope object with `schema_version`, `trip_id`, and `issues` (breaking change from the v0.9.3 array format).
- **Stable issue fields**: `kind` → `code` (snake_case), `target` as `{ "type": "...", "id": ... }`, and typed `details` for machine-readable metadata.
- **`trip advisor --json`**: New JSON output with the same issue structure plus `advice` and optional `commands` (`--with-commands`).
- **Text output unchanged**: Human-readable doctor/advisor output is the same as before.

## Breaking changes

Prior `trip doctor --json` consumers must migrate:

| Before (v0.9.3) | After (v1.0.6) |
|---|---|
| Root: JSON array | Root: object with `schema_version`, `trip_id`, `issues` |
| `kind`: PascalCase enum tag | `code`: snake_case string |
| `target`: `{"Day": 3}` style | `target`: `{ "type": "day", "id": 3 }` |
| No `details` | `details`: typed fields per issue code |

## `target.id` semantics

| `target.type` | Meaning of `target.id` |
|---|---|
| `trip` | Trip ID (`trips.id`) |
| `day` | Day number within the trip (1-based; not a DB row ID) |
| `itinerary` | Itinerary item ID (`itinerary_items.id`) |

## `details` fields by code

| `code` | `details` fields |
|---|---|
| `empty_itinerary` | (omitted) |
| `overloaded_day` | `day`, `itinerary_count` |
| `no_restaurant` | `day` |
| `high_travel_time` | `day`, `travel_minutes` |
| `missing_duration` | `itinerary_id` |

## Advisor JSON

`trip advisor --json` adds:

- `with_commands`: whether `--with-commands` was passed
- Per issue: same fields as doctor JSON, plus `advice[]` and `commands[]` (commands omitted when empty)

Example:

```bash
cargo run -- trip advisor 1 --json --with-commands
```
