# v3.6.0 — Receipt Inbox metadata-only support

## Summary

**v3.6.0** implements **Receipt Inbox metadata-only support** — Trip-level storage for unreviewed payment candidates waiting to be reviewed, ignored, or eventually promoted to an Expense in future work. This is an **implementation release** (not documentation-only).

Requires **v3.5.0** baseline. Current trip export now writes **`schema_version: 7`**.

### Terminology

| Term | Meaning |
|---|---|
| **Estimate** | Planned amount before the trip |
| **Receipt** | Unreviewed payment candidate waiting for Expense promotion (not Actual) |
| **Expense** | Confirmed actual spending |
| **Difference** | Planned-vs-actual delta based on **confirmed** Expenses |

---

## Added

- Added metadata-only Receipt Inbox support.
- Added `receipts` table as a **Trip-level collection** (not nested under Itineraries).
- Added receipt commands:
  - `receipt add`
  - `receipt list`
  - `receipt show`
  - `receipt update`
  - `receipt ignore`
  - `receipt delete`
- Added JSON output support for `receipt list` / `receipt show` (`--json`).
- Added export schema **v7** with top-level `receipts[]`.
- Added import support for schema v7 receipts.
- Added `validate-export` checks for receipts.
- Added `trip diff` support for receipt additions, removals, and modifications.

---

## Changed

- Export schema version is now **v7**.
- Receipt lifecycle (simplified before release):
  - `unreviewed` — not yet promoted to Expense
  - `ignored` — not a travel expense, duplicate, or out of scope
- Receipts are treated as **Expense candidates**, not confirmed Expenses.
- Receipts remain **Trip-level records** and are not nested under `days[].itineraries[]`.
- Optional **Day reference** (`day_id` / `day_ref`) is supported as a hint.
- Itinerary reference and linked Expense reference are **intentionally not part** of the released model.
- `receipt link` is **not** included (removed before release).

---

## Compatibility

- **v6 imports** remain supported (`receipts` omitted = empty).
- Exports without receipts remain importable.
- Existing **Planned vs Actual** behavior remains unchanged.
- Existing DBs receive `receipts` table migration on first run after upgrade.
- Legacy receipt schema with `itinerary_id` / `linked` / `converted` is normalized on migration.

## Notes

Receipt Inbox is intentionally **metadata-only** in v3.6.0.

The released model treats Receipt as an unreviewed payment candidate waiting to be reviewed, ignored, or eventually promoted to an Expense in future work. A Receipt is **not** an Actual Expense and is **not** included in Planned vs Actual calculations.

Image evidence should be designed as a shared **Evidence / Attachment** layer for both Receipt and Expense in future work — not as Receipt-only `image_path`.

---

## Upgrade

1. Replace binary with v3.6.0 build (or `cargo build --release` from tag `v3.6.0`).
2. On first run, `receipts` table is created automatically (`migrate_receipts`). No manual migration step.
3. `trip export` now writes `schema_version: 7` with optional `receipts[]`.
4. v6 export files import without change.
