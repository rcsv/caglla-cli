# v4.6.3 — Command handler split Phase 1

## Summary

**v4.6.3** is a **documentation-first** review for splitting CLI command handlers from reusable core/service logic, in preparation for future Tauri GUI.

Review: [v4.6.3-command-handler-split-phase-1.md](../specifications/v4.6.3-command-handler-split-phase-1.md)

## Headline findings

- **~1,078-line `main.rs`** holds most command match arms; only `db` and `about` live under `commands/`
- Domain modules (`trip.rs`, `itinerary.rs`, etc.) mix **repository + terminal `print_*` + use case** logic
- **No explicit service layer** — GUI cannot reuse handlers without pulling in `println!`
- **Read-only commands first** — `trip list/show`, `day list`, `itinerary timeline`, `trip stats` are safest split pilots

## Recommended approach

```text
CLI layer        → clap, human/JSON output, exit codes
Service layer    → use cases, structured results (no println)
Repository layer → SQLite
Domain layer     → types and rules

Tauri → same service layer as CLI
```

## Non-goals

- Tauri / GUI implementation
- Large-scale handler moves
- DB schema / migration changes
- CLI behavior or golden changes
