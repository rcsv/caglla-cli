# v4.6.11 — Read-only service boundary review

## Summary

**v4.6.11** is a **documentation-only review** after v4.6.4〜v4.6.10 read-only service boundary rollout. It inventories the seven service modules, confirms CLI/service/repository boundaries, assesses GUI-first read-only coverage, and recommends the next implementation step.

Review: [v4.6.11-read-only-service-boundary-review.md](../specifications/v4.6.11-read-only-service-boundary-review.md)

## Headline findings

- **7 service modules** — `trip_stats`, `trip_list`, `trip_show`, `day_list`, `itinerary_list`, `itinerary_timeline`, `itinerary_show`
- **No terminal I/O in services** — `println!` / `eprintln!` absent from `src/services/`
- **JSON shape preserved** — CLI serializes existing domain types; wrappers stay in CLI layer
- **GUI primary path ready** — Trip list → show/stats → day list → itinerary list/timeline → itinerary show
- **Recommended next** — **v4.6.12 `note list` service boundary**

## Key design decisions (confirmed)

| Decision | Rationale |
|---|---|
| Reservations on `itinerary show` stay in CLI | JSON is `ItineraryItem` only; human-only section |
| `DayListJson` built in CLI | Service returns `Trip` + `Vec<Day>` |
| No `--json` on `itinerary timeline` | Existing human-only command; service result for future GUI |
| No JSON wrappers in service results | Compatibility with existing CLI `--json` shapes |

## Non-goals

- New service boundary implementation
- Rust functional changes
- CLI / JSON output changes
- DB schema / migration changes
