# v3.5.0 — Receipt Inbox concept design

## Summary

**v3.5.0** is a **documentation-only release**. It adopts [Receipt Inbox concept design](../specifications/v3.5.0-receipt-inbox-concept-design.md) as the formal v3.5.0 design document. No CLI behavior, DB schema, or export/import schema changes.

Requires **v3.4.0** baseline. Current trip export still writes **`schema_version: 6`**.

Design series:

```text
Concept Design                 → v3.5.0-receipt-inbox-concept-design.md
Release v3.5.0                 → this release (documentation-only)
```

### Terminology

| Term | Meaning |
|---|---|
| **Estimate** | Planned amount before the trip |
| **Receipt** | Unreviewed payment evidence |
| **Expense** | Confirmed actual spending |
| **Difference** | Planned-vs-actual delta based on **confirmed** Expenses |

---

## Added

- Added Receipt Inbox concept design as a documentation-only release.
- Defined Receipt as unreviewed payment evidence that can exist before it is linked to an itinerary or converted into an Expense.
- Clarified the responsibility split:
  - Estimate = planned amount
  - Receipt = unreviewed evidence
  - Expense = confirmed actual spending
  - Difference = planned-vs-actual delta based on confirmed Expenses
- Positioned Receipt Inbox as post-trip review support, not during-trip budget control.

---

## Clarified

- Receipt Inbox should not immediately affect Planned vs Actual difference.
- Unreviewed receipts may be shown as auxiliary information in future post-trip review flows.
- Receipt images are payment evidence and should remain separate from generic Photo / Attachment modeling.
- Expense should remain a confirmed actual spending record, not a catch-all inbox.
- Itinerary should not be used as a dummy container for unreviewed receipts.
