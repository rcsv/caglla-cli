# v1.18.0 — Reservation implementation

## Summary

Implements **Reservation** — structured booking and confirmation information tied to Itinerary actions.

Design series (completed before this release):

```text
v1.11.0  Responsibilities Review (reservation-model)
v1.12.0  Entity Design
v1.13.0  Implementation Plan
v1.18.0  Implementation  ← this release
```

### Model

```text
保存: Itinerary 配下（reservations.itinerary_id）
表示: Trip 単位で集約（reservation list --trip / export-md）
```

Reservation is **not** a Trip-level contract ledger. It is booking/confirmation data for executing an Itinerary action.

## Added

### Database

New table `reservations`:

| Column | Notes |
|---|---|
| `itinerary_id` | NOT NULL — parent Itinerary (app-side cascade on delete) |
| `reservation_type` | Required — `hotel`, `flight`, `restaurant`, `rental_car`, `activity`, `parking`, `ticket`, `other` |
| `provider_name` | Required |
| `confirmation_code` | Optional |
| `reservation_site_url` | Optional |
| `remark` | Optional |
| `start_at` / `end_at` | Optional — contract period (hotel, rental car, etc.) |

- Existing DBs: `CREATE TABLE` migration only — no backfill
- Index: `idx_reservations_itinerary`

### CLI

| Command | Behavior |
|---|---|
| `reservation add` | `--itinerary` required; `--reservation-type`, `--provider` required |
| `reservation list` | `--itinerary` or `--trip` (Trip-level aggregation) |
| `reservation show` | By ID; `--json` supported |
| `reservation update` | By ID |
| `reservation delete` | By ID |

`itinerary show` displays linked Reservations inline when present.

### Export / import (schema v3)

Reservations are nested under itineraries (not top-level):

```json
{
  "days": [
    {
      "itineraries": [
        {
          "reservations": []
        }
      ]
    }
  ]
}
```

- Empty `reservations` arrays are omitted on export
- Legacy v3 import without `reservations` keys → empty (backward compatible)
- `trip duplicate` preserves reservations via export/import roundtrip
- `validate-export` validates reservation fields

### Markdown export

`trip export-md` adds a **`## Reservations`** section when any exist:

- Grouped by reservation type (e.g. `### Hotel`)
- Shows Day / Itinerary context, provider, confirmation, URL, remark, period

### Diff

`trip diff` detects:

```text
+ Reservation added
- Reservation removed
~ Reservation modified
```

Comparison key: Itinerary context + `reservation_type` + `provider_name` + `confirmation_code`.

## Validation

| Rule | Behavior |
|---|---|
| `itinerary_id` | Must exist |
| `reservation_type` | Required; must be a known type |
| `provider_name` | Required (non-empty after trim) |
| Optional text fields | Trim; empty → `NULL` |
