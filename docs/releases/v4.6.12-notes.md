# v4.6.12 — Note list service boundary

## Summary

**v4.6.12** introduces a minimal **service layer** for the read-only `note list` command, keeping existing CLI human output, JSON shape (`NoteListJson`), note ordering, and owner-resolution / not-found behavior unchanged while making the use case reusable for future Tauri GUI detail screens.

Specification: [v4.6.12-note-list-service-boundary.md](../specifications/v4.6.12-note-list-service-boundary.md)

## What changed

- New `src/services/note_list.rs` — `list_notes()` returns `NoteListServiceResult` (no terminal I/O)
- `note list` CLI handler calls service, then routes to existing human/JSON output

## Compatibility

- `note list` human output unchanged (Trip / Day / Itinerary targets)
- `note list --json` unchanged (`NoteListJson { owner_type, owner_id, notes }`)
- Note ordering unchanged (`ORDER BY sort_order ASC, id ASC`)
- Owner resolution and not-found messages unchanged
- No DB schema or migration changes
- Golden updates: Okinawa Markdown Colophon `Version:` only

## Non-goals

- No `note show` / write command changes
- No `expense list` / `reservation list` / `checklist list` service split
- No Tauri / GUI implementation
