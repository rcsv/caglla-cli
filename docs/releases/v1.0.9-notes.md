# v1.0.9 Day model and required trip dates

## Summary

- Trips now require `start_date` and `end_date` on create and update.
- Introduced the `Day` entity with automatic generation from the trip date range.
- CLI UX is unchanged: `itinerary add --day N` still works; `day_id` is not exposed yet.

## Highlights

- **`trip add`** requires `--start` and `--end` (`end_date >= start_date`)
- **Day model** — Trip → Day → Itinerary (internal); `days` table stores `day_number`, `title`, `description`
- **Day auto-generation** on trip create (Day 1..N from inclusive date range)
- **Day sync on trip update** — extend adds days; shrink removes empty days only (errors if itinerary or day metadata exists)
- **Day dates derived from Trip** — `trip.start_date + (day_number - 1)`; no `date` column on `days`
- **Documentation** — README updated; [`docs/specifications/day-model.md`](../specifications/day-model.md) added

## Technical Changes

### New module and schema

- `src/day.rs` — date validation, day CRUD helpers, sync on trip date changes
- `days` table:

```sql
CREATE TABLE days (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  trip_id INTEGER NOT NULL,
  day_number INTEGER NOT NULL,
  title TEXT NOT NULL DEFAULT '',
  description TEXT,
  ...
  UNIQUE(trip_id, day_number)
);
```

- `migrate_days` backfills existing trips (date range or max itinerary day)
- `reset_db` clears `days` before `trips`

### Trip behavior

- `add_trip(name, start, end)` validates range and creates Day 1..N
- `update_trip` requires both dates after merge; calls `sync_days_to_trip_duration`
- Import requires `trip.start_date` and `trip.end_date`; creates days on import

### Not in this release

- `itinerary_items.day_id` (still uses `day` integer)
- export schema v2 (`days[]`)
- `day` CLI commands / swap
- Markdown Day title/description sections

## Breaking Changes

| Before | After |
|---|---|
| `trip add "Name"` (dates optional) | `trip add "Name" --start YYYY-MM-DD --end YYYY-MM-DD` required |
| Legacy export JSON without dates importable | import fails without `trip.start_date` / `trip.end_date` |
| No `days` table | `days` rows created for every new trip |

Existing DBs: run the app once to apply `migrate_days` backfill, or use `db reset` in development.

## Compatibility

- `itinerary add --day N` unchanged for users
- export JSON schema v1 unchanged (no `days[]` yet)
- `itinerary_items.day` column retained
