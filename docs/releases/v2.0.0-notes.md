# v2.0.0 — Participant Foundation

## Summary

First **v2** feature release. Implements **Participant** — Trip-scoped participation records for who is traveling on a trip, including a self marker (`is_self`), CRUD CLI, export **schema v4**, and integration with diff / Markdown export / stats / doctor / advisor.


### Terminology

| Term | Meaning |
|---|---|
| **Participant** | Everyone traveling on the Trip — **including yourself** |
| **Companion** | Everyone except yourself |
| **Participation record** | Trip-scoped row in `participants` — **not** a global Person profile |

### Count semantics

```text
participant_count / traveler_count = count(participants)   when self is known
companion_count = count(participants) - 1                  only when exactly one is_self = true
self unknown or participants not recorded                  → companion_count is unknown (not 0)
```

## Added

### Database

New table `participants`:

| Column | Notes |
|---|---|
| `trip_id` | NOT NULL — Trip scope |
| `name` | NOT NULL — display name (trimmed, max 200 chars) |
| `sort_order` | NOT NULL — ordering within Trip (default: max + 1 on add) |
| `is_self` | NOT NULL DEFAULT 0 — marks **you** on this Trip (max 1 per Trip) |
| `created_at` / `updated_at` | timestamps |

Indexes:

- `idx_participants_trip` on `trip_id`
- `idx_participants_one_self_per_trip` — **partial unique** on `trip_id` WHERE `is_self = 1`

Existing DBs: `migrate_participants` on `open_db` — idempotent `CREATE TABLE IF NOT EXISTS`.

### CLI

| Command | Behavior |
|---|---|
| `participant add --trip <id> --name <name> [--sort-order N] [--self]` | Add participant; `--self` errors if self already exists |
| `participant list --trip <id> [--json]` | List with counts (`participant_count`, `companion_count`, `self_known`) |
| `participant show <id> [--json]` | Detail by participant ID |
| `participant update <id> [--name] [--sort-order] [--self \| --not-self]` | Update; `--self` transfers self marker from previous row |
| `participant delete <id>` | Delete (self removal allowed; stats then treat self as unknown) |

### Export / import (schema v4)

Current `trip export` writes **`schema_version: 4`** with top-level `participants[]`:

```json
{
  "schema_version": 4,
  "trip": { "name": "沖縄旅行" },
  "participants": [
    { "name": "ともさん", "sort_order": 0, "is_self": true },
    { "name": "妻", "sort_order": 1, "is_self": false }
  ],
  "days": []
}
```

- Internal participant `id` is **not** exported
- `is_self` is exported
- **v3 import** remains supported (`participants` omitted → empty)
- **v4 import** restores participants after Trip creation
- `validate-export` rejects multiple `is_self: true` in one Trip
- `trip duplicate` copies participants via export/import roundtrip
- `db reset` clears `participants` (fixes import roundtrip after reset)

### Diff

`trip diff` reports Participant changes:

- added / removed
- `is_self` changed (identity key: `sort_order` + `name`)
- rename / reorder detected conservatively as removed + added

### Markdown export

`trip export-md` adds **`## Participants`** after Overview when any exist (table with Self yes/no). Not shown under Day / Itinerary.

### Stats / doctor / advisor

- **trip stats:** `participant_count`, `traveler_count`, `companion_count`, `self_known`, `participants_recorded`
- **doctor / advisor (info):** `PARTICIPANTS_NOT_RECORDED`, `SELF_PARTICIPANT_UNKNOWN`
- **doctor (warning):** `MULTIPLE_SELF_PARTICIPANTS` (data error)

Unregistered solo trips show **"not recorded"** — not "0 travelers".

## Migration / compatibility

| Topic | Notes |
|---|---|
| **Existing `caglla.db`** | Opening the DB runs `migrate_participants` — no manual step |
| **Export v3 files** | Import still works; participants default to empty |
| **Export v4** | Use v2.0.0+ CLI to export/import with participants |
| **Downgrade** | v1.22 CLI cannot read v4-only participant data in DB; export v4 JSON before downgrade |

## Upgrade

1. Replace binary with v2.0.0 build (or `cargo build --release` from tag `v2.0.0`).
2. Run any command — DB migration applies automatically.
3. Optional: `participant add --trip <id> --name "Your name" --self` for existing trips.
4. Re-export trips to obtain schema v4 JSON with `participants[]`.
