# v1.8.0 — Itinerary model (not a venue)

## Summary

v1.8.0 is a **documentation / specification release**. No implementation or DB schema changes.

The main deliverable is [`docs/specifications/itinerary-model.md`](../specifications/itinerary-model.md), which defines the CLI Itinerary model as a **unit of travel activity**, not a venue.

## Design principles

| Principle | Meaning |
|---|---|
| **Itinerary is not a venue** | An itinerary is not a POI or place record |
| **Itinerary is a unit of travel activity** | Plans before the trip, records after — one row per action in the timeline |
| **Place / Venue is optional** | `location` is a free-text optional field; no `place_id` is required |
| **Expense belongs under Itinerary** | Costs are recorded as Expense children, not on the Itinerary row itself |
| **note vs Note** | `itinerary_items.note` for short notes; Note entity for longer / multiple memos |
| **category is auxiliary** | Used for display, aggregation, and checklist generation — not required to create an itinerary |
| **Ordering** | `start_time` and `sort_order` express sequence within a Day |

### Real-world activities (canonical sample)

Using [`samples/okinawa_sesoko_2026/`](../../samples/okinawa_sesoko_2026/) as evidence, the spec explains why non-POI items are valid itineraries:

- Departure, highway, parking, airport check-in
- Grocery shopping, in-room dinner, hotel check-in / check-out
- Ferry boarding, refueling, rental car return, return home

These are ordered travel activities — anchors for expenses, notes, checklists, and future Note / Photo attachments.

### PDF / Excel mapping

- **One spreadsheet row ≠ one Itinerary**
- Schedule lines → Itinerary
- Amount / receipt lines → Expense under the matching Itinerary
- Multiple receipts for the same action → **one Itinerary, multiple Expenses** (e.g. souvenir shop with four separate receipts)
