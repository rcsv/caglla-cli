# v1.11.0 — Reservation model specification

## Summary

Documentation-only release. Adds the **Reservation Model Specification** and clarifies how Reservation fits in the Travel Ledger model alongside Itinerary, Expense, Remark, Note, and future Venue.

Reservation is defined as **booking and confirmation information required to execute a travel activity** — not a Note, not an Expense, not a Venue.

## Added

- Added `docs/specifications/reservation-model.md` — Reservation entity design draft.
- Updated specification index and cross-links from travel-ledger-responsibilities, note-model, and itinerary-model.

## Clarified

| Topic | v1.11.0 documentation |
|---|---|
| **Reservation vs Note** | Structured booking data vs long-form free text |
| **Reservation vs Expense** | Booking/confirmation vs cost records (both can exist on one Itinerary) |
| **Reservation vs Remark** | Full reservation entity vs short inline `itinerary_items.note` |
| **Itinerary placement** | Reservation belongs under Itinerary; Trip-level aggregation for guidebook export |
| **Temporal span** | Reservations can express intervals (e.g. check_in/check_out, pickup/return) |
| **Routing** | **Out of scope** — spatial movement routes are a future independent model |
| **Design principle** | Do not expose generic relationship modeling to users; operations remain **itinerary-centric** |
