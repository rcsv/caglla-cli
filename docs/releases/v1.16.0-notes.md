## Clarified

### Implementation goals

Add `Trip.summary` and `Day.summary` for guidebook-style sharing; wire through CLI display, JSON export/import, Markdown export, and `trip diff`.

### Trip / Day scope

| Level | Plan |
|---|---|
| **Trip.summary** | New `trips.summary TEXT NULL` column |
| **Day.summary** | Day-level short overview — **existing `days.description` must be reconciled** (see below) |
| **Itinerary.summary** | **Not in first cut** — `itinerary.title` remains the action label |

### Existing `days.description`

The current DB already has `days.description` (unused in export v3). The implementation plan documents three future options:

| Option | Approach |
|---|---|
| **A (recommended)** | Rename `description` → `summary` |
| **B** | Keep column name; assign Summary semantics; export key `summary` |
| **C** | Add new `summary` column alongside `description` |

Decision deferred to the implementation phase.

### Implementation areas (future — documented only)

| Area | Plan summary |
|---|---|
| **DB migration** | `trips.summary` add; Day side per option above; NULL default; no backfill; empty string → NULL at app layer |
| **CLI** | `trip add/update/show --summary`; new `day update --summary`; `--clear-summary`; list commands omit summary initially |
| **Export / import** | v3 extension candidate (`trip.summary`, `days[].summary`); v4 re-evaluation possible |
| **Markdown** | Trip title and Day heading; plain text; omit when NULL |
| **Diff** | Compare `trip.summary` and per-day summaries |
| **Validation** | trim, empty → NULL, length limits, Note redirect message |
| **Tests** | migration, CLI, roundtrip, export-md, diff, legacy import |

### Non-goals (this plan)

Itinerary.summary, separate Summary table, rich text, Reservation / Participant / Photo integration, canonical sample updates.
