# v1.1.0 Itinerary day_id model

## Summary

- Itinerary items now reference `days.id` via `day_id` (Day model completion).
- CLI UX is unchanged: `itinerary add --day N` and `itinerary update --day N` still use day numbers.
- Export/import remain on schema v1 (`itinerary_items[].day` as day_number).

## Highlights

- **`day_id` foreign key** — `itinerary_items.day_id` → `days(id)`
- **Day resolution helpers** — `find_day_id_by_trip_and_day_number`, `find_day_by_trip_and_day_number`
- **Migration** — backfill `day_id` from `trip_id` + legacy `day` column; fail if unresolved rows remain
- **Indexes** — `idx_itinerary_items_day_id`, `idx_itinerary_items_trip_id`, `idx_days_trip_day_number`
- **CLI compatibility** — `itinerary list`, `timeline`, `trip stats`, export/import behave as before
- **`day` column retained** — denormalized day_number for export v1; removal deferred to a future release

## Technical Changes

### Database

```sql
ALTER TABLE itinerary_items ADD COLUMN day_id INTEGER REFERENCES days(id);
```

- `migrate_itinerary_day_id` runs after `migrate_days`
- Reads use `INNER JOIN days d ON i.day_id = d.id` to populate `ItineraryItem.day` (day_number)
- Writes resolve `--day N` to `day_id` via `find_day_id_by_trip_and_day_number`

### Not in this release

- Day CRUD / swap commands
- export schema v2 (`days[]`)
- Markdown Day title/description sections
- `itinerary_items.day` column removal
- Performance caching or precomputed stats columns

## Breaking Changes

None for CLI users. Existing databases are migrated automatically on startup (`migrate_itinerary_day_id`).

## Compatibility

| Area | Behavior |
|---|---|
| CLI `--day N` | Unchanged |
| export JSON schema v1 | Unchanged (`day` integer in JSON) |
| import | Unchanged; import sets `day_id` after creating days |
| stats / timeline | SQL aggregation via joined queries; same output |
