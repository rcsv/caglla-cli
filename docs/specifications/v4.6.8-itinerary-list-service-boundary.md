# Itinerary list service boundary

v4.6.4〜v4.6.7 の read-only service boundary に続き、**`itinerary list` 単独**で service 層を導入する。

| Item | Status |
|---|---|
| Target | v4.6.8 — itinerary list service boundary |
| Target command | **`itinerary list` のみ** |
| Scope | service 層導入（structured result）、CLI 出力互換維持 |

---

## Status

| 観点 | 判定 |
|---|---|
| service layer | **`src/services/itinerary_list.rs`** 追加 |
| target command | `itinerary list` が service 経由 |
| terminal I/O | service 内に `println!` / `eprintln!` なし |
| CLI / JSON 出力 | **不変** |
| Ordering | **不変**（Sequence-first: Day → `sort_order` → `id`） |
| Trip not found | **不変**（`Trip not found: <id>`） |
| DB schema / migration | **変更なし** |

---

## Background

目標形:

```text
CLI command
  → service/usecase
  → repository
  → SQLite

Tauri command
  → same service/usecase
  → repository
  → SQLite
```

`itinerary list` は GUI の Day overview → Itinerary list 導線に直結する。v4.6.8 では `itinerary timeline/show/add/update/delete` 等には触れない。

---

## Target command

- `itinerary list <trip_id>`
- `itinerary list <trip_id> --json`

---

## Current flow before split

```text
main.rs (ItineraryAction::List)
  → itinerary::list_itinerary_items(conn, trip_id) -> Vec<ItineraryItem>
  ├─ --json  → print_json(&Vec<ItineraryItem>)
  └─ human   → println!("旅行 ID ...") + print_itinerary_list(&Vec<ItineraryItem>)
```

---

## Service boundary introduced

新規:

```text
src/services/itinerary_list.rs
```

API:

```rust
pub struct ItineraryListServiceResult {
    pub items: Vec<ItineraryItem>,
}

pub fn list_itineraries(conn: &Connection, trip_id: i64) -> Result<ItineraryListServiceResult>
```

service は `itinerary::list_itinerary_items` を呼び、**structured result のみ**返す（terminal I/O なし）。

---

## CLI compatibility

- human 出力は従来どおり `println!("旅行 ID ...")` + `print_itinerary_list` を使用
- empty state も `print_itinerary_list` に委譲（`日程はまだ登録されていません。`）

---

## JSON compatibility

- `itinerary list --json` は引き続き `Vec<ItineraryItem>` を serialize（配列 shape / field は不変）

---

## Ordering compatibility

Sequence-first ordering は従来どおり `itinerary::list_itinerary_items` の SQL で担保される:

```text
ORDER BY d.day_number, i.sort_order, i.id
```

service は順序を変えない。

---

## Error compatibility

- Trip 不存在: `Trip not found: <id>`（`itinerary::list_itinerary_items` が `trip::get_trip` を呼ぶ）

---

## Test coverage

- service test:
  - Trip not found message を維持
  - Day → sort_order → id の ordering を維持
- 既存 CLI not-found テスト（`tests/not_found_cli.rs`）を回帰として維持

---

## What changed

- `src/services/itinerary_list.rs` 追加
- `src/services/mod.rs` に `itinerary_list` module 追加
- `main.rs` の `ItineraryAction::List` が service 経由に

---

## What did not change

- `itinerary show` / `itinerary timeline` / `itinerary add/update/delete`（対象外）
- `day list` / `trip list` / `trip show` / `trip stats`（対象外）
- write command / migration / FK（変更なし）
- DB schema / JSON schema（変更なし）

---

## Lessons learned

- `itinerary list` は既に `Vec<ItineraryItem>` を返しており、service は thin wrapper で十分
- ordering は repository 側（SQL）で担保されているため、service は加工しない方が安全
