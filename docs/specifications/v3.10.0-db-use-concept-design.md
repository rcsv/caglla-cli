## 1. 背景

v3.9.0 Phase 1 では DB path を **読み取り専用** で解決できるようになった。

```text
1. --db <path>
2. CAGLLA_DB
3. ./caglla.toml [database].path
4. ./caglla.db（default）
```

ユーザーは `--db` や `CAGLLA_DB` で都度指定できるが、**プロジェクトごとに既定 DB を記録する手段** がない。毎回フラグや環境変数を渡すのは、サンプル DB・本番 DB・テスト DB の混同防止という v3.2.0 の目的に対して不便である。

Phase 2 の **`db use`** は、CWD の `./caglla.toml` に `[database].path` を **書き込む** ことで、以降のコマンド（override なし）が同じ DB を参照するようにする。

---

## 2. 問題意識

| 現状（v3.9.x） | 困りごと |
|---|---|
| `caglla.toml` は手動編集のみ | エディタを開かずに切り替えたい |
| `--db` は毎回指定 | シェル履歴・エイリアスに依存 |
| `db path` は read のみ | 「今後この DB を使う」と宣言するコマンドがない |
| リポジトリ直下の `./caglla.db` 固定運用が多い | `./data/okinawa.db` 等をチームで共有しづらい |

`db use` は **「今このコマンドだけ DB を変える」** ための `--db` ではなく、**「このディレクトリで作業するときの既定 DB を config に記録する」** ためのコマンドである。

---

## 3. `db use` の目的

```text
CWD の ./caglla.toml に [database].path を設定し、
--db / CAGLLA_DB が無い通常コマンドが参照する DB を永続化する。
```

### 3.1 やること

- `./caglla.toml` を作成または更新する
- `[database].path` のみを対象とする（Phase 2 実装スコープ）
- DB ファイル自体は **作成しない**（`db path` / 未存在時の `db status` と同様の原則）
- 成功時に human 向けに **書き込んだ path** と **config ファイル path** を表示する

### 3.2 やらないこと（Phase 2 実装でも）

```text
- 現コマンド実行中の DB を --db のように即 override しない（書き込みのみ）
- 親ディレクトリの caglla.toml 探索
- ユーザーグローバル config（~/.config/caglla/config.toml 等）
- profile / 複数 DB 切替
- remote DB / cloud sync
- SQLite schema / trip export schema 変更
```

---

## 4. 解決優先順位（変更なし）

`db use` 実装後も **read resolution** は v3.9.0 と同じ。

```text
1. --db <path>          ← db use より常に優先（当該 invocation のみ）
2. CAGLLA_DB            ← 同上
3. ./caglla.toml [database].path   ← db use が更新する層
4. ./caglla.db
```

**重要:** `db use ./foo.db` を実行した **同じ invocation** で続けて trip 等を走らせるわけではない（サブコマンドが 1 つの CLI 設計）。ユーザーが次に `caglla trip list` を実行したとき、config 経由で `foo.db` が選ばれる。

## 5. コマンド仕様（設計）

### 5.1 基本形

```bash
caglla db use <path>
caglla db use ./data/okinawa.db
caglla db use /abs/path/to/project.db
```

| 項目 | 設計 |
|---|---|
| `<path>` | 必須。CWD 基準で解釈し、config に保存する **文字列** を正規化（§6） |
| DB open | **しない** |
| DB 作成 | **しない** |
| 対象 config | **CWD の `./caglla.toml` のみ** |

### 5.2 `db use --clear`

**採用する。** 既定 DB（`./caglla.db`）に戻す明示的操作として必要。

```bash
caglla db use --clear
```

| 項目 | 設計 |
|---|---|
| 効果 | `[database].path` を削除 |
| `[database]` が空 | `[database]` セクションごと削除 |
| 他セクションなし | **`caglla.toml` ファイルごと削除**（存在していた場合） |
| 効果（resolution） | 以降は priority 4 の default `./caglla.db` |

`--clear` と `<path>` は **同時指定不可**（clap 相互排他）。

### 5.3 出力（human、案）

```text
Database path saved to config
  Config : /abs/cwd/caglla.toml
  Path   : ./data/okinawa.db
```

`--clear` 時:

```text
Database path cleared from config
  Config : /abs/cwd/caglla.toml
  (default ./caglla.db will be used when no --db or CAGLLA_DB)
```

ファイル削除時は `Config : (removed)` 等で明示。

JSON 出力は Phase 2 実装時に **任意**（Phase 1 の `db status` に合わせ `--json` を付けるかは Implementation Plan で決定）。

## 6. パス保存ルール

### 6.1 原則

**portable な相対パスを優先**し、必要な場合のみ絶対パスを保存する。

| ユーザー入力 | 保存値（`[database].path`） | 理由 |
|---|---|---|
| `./data/app.db` | `./data/app.db` | config dir 基準・リポジトリ内で共有可能 |
| `data/app.db` | `./data/app.db`（正規化） | 同上 |
| `/Users/me/project/app.db` | 下記参照 | 絶対パス入力。config dir との位置関係により相対化可否を判断 |
| `../shared/other.db` | `../shared/other.db` | `..` を含む相対は許可（Phase 1 resolver と同じ解決） |

絶対パスが与えられた場合は、config dir からの相対化を試みる。

- config dir 配下、または相対表現として安全に保存できる場合 → `./app.db` などの相対パス
- config dir 外、別ドライブ、または相対化できない場合 → 絶対パス

### 6.2 正規化

- Phase 1 の `normalize_path`（`.` セグメント除去）と整合させる
- config **ファイルのディレクトリ** を基準に、保存前に「config dir からの相対表現にできるか」を判定
- できる場合 → `./` 付き相対（または `..` 始まり）で保存
- できない場合（別ドライブ等）→ 絶対パスで保存

### 6.3 読み取りとの整合

Phase 1 どおり、config 内の相対 path は **config ファイルのディレクトリ** 基準で解決する。保存形式と read 側は対称である。

## 7. `caglla.toml` 更新ルール

### 7.1 書き込み先

```text
Phase 2: CWD / caglla.toml のみ
```

親ディレクトリ探索は **Phase 2 に含めない**（§10）。

### 7.2 ファイルが存在しない

最小内容で **新規作成**:

```toml
[database]
path = "./data/okinawa.db"
```

### 7.3 ファイルが存在する

1. TOML として parse 可能であること（Phase 1 と同様、invalid はエラー）
2. **`[database].path` のみ更新**（merge）
3. **他の top-level キー・テーブルは保持**（例: 将来の `[profile]` 等）

### 7.4 未知キー

| 扱い | 設計 |
|---|---|
| 未知の top-level テーブル | **保持**（parse → merge → serialize） |
| 未知の `[database]` 下キー | **保持**（`path` だけ上書き） |
| invalid TOML | **エラー**（ファイルを壊さない — 書き込み前に検証） |

### 7.5 コメント・空行

Phase 2 実装の **既知の制限** として明記する。

| 方式 | Phase 2 推奨 |
|---|---|
| TOML crate による parse + serialize | **採用** — 実装が単純 |
| コメント・空行 | **保証しない**（round-trip で失われる可能性） |
| ユーザー向け | `db use` 前に手動編集したコメントは消える可能性がある旨を help / docs に記載 |

コメント完全保持が必要になった場合は、Phase 3 以降で surgical edit（行指向更新）を検討。**Phase 2 ではスコープ外。**

### 7.6 アトミック書き込み

Implementation Plan で以下を要求する（設計上の非機能要件）:

- 一時ファイルへ write → rename（同一ディレクトリ）
- 失敗時は既存 `caglla.toml` を残す

## 8. DB ファイル未存在 path

**許可する。**

| コマンド | 未存在 path |
|---|---|
| `db use <path>` | **成功**（config 更新のみ） |
| `db path` / `db status` | 従来どおり open しない / exists: no |
| 通常コマンド | 初回 open 時に作成（v3.9.0 同様） |

任意: human 出力に `Note: database file does not exist yet` を付ける（warning 程度、exit 0）。

**理由:** ユーザーは「これから使う DB」を先に config に登録してから `trip add` 等で初期化する流れが自然。`--db ./new.db` が未存在でも trip コマンドは動くのと整合する。

---

## 9. 他 `db` サブコマンドとの関係

| コマンド | `db use` 後の挙動 |
|---|---|
| `db path` | 更新後の config を反映した resolved path（override なし時） |
| `db status` | 同上。未存在なら作成しない |
| `db reset` | resolved DB に対して reset（従来どおり open 後） |

`db use` は **`run_before_open_db` 相当で完結** し、DB を open しない（`db path` と同系）。

---

## 10. Phase 境界 — 親ディレクトリ探索

**Phase 2（`db use` 実装）には含めない。** さらに **Phase 3 としても優先度は低** — 別 milestone とする。

| 機能 | 推奨 milestone |
|---|---|
| `db use` + CWD `./caglla.toml` 書き込み | **Phase 2**（v3.11.x 実装候補） |
| 親ディレクトリへ向かう config 読取探索 | **Phase 3+**（git ルート自動検出等 — 要別 design） |
| user-global config | **defer**（profile / Identity 連動候補） |
| profile 切替 | **defer** |

**理由:** 親 dir 探索を入れると **read** と **write** の両方のルールが変わる（どの `caglla.toml` を `db use` が更新するか）。Phase 2 は read（v3.9.0）と write（CWD 固定）を対称に保つのが最小。

## 11. 実装 milestone（参考 — 本 release では未着手）

```text
v3.10.0  Concept Design（本書）     ← documentation-only
v3.11.x  Implementation Plan + db use 実装（別判断）
         - write_config_path / db use / db use --clear
         - merge TOML、atomic write
         - tests/db_cli.rs integration
```

Implementation Plan 作成時のテスト候補:

- 新規 `caglla.toml` 作成
- 既存 `[database].path` 上書き
- 未知 top-level キー保持
- `--clear` で default に戻る
- 未存在 DB path で `db use` 成功
- `db use` 後の `db path` が config source

## 13. 設計判断サマリー

| 論点 | 決定 |
|---|---|
| `db use` の責務 | `./caglla.toml` の `[database].path` を更新 |
| 即時 DB 切替 | **しない** — 次回以降の config source を設定 |
| `--db` / `CAGLLA_DB` | config より **優先**（変更なし） |
| 書き込み先 | **CWD `./caglla.toml` のみ** |
| 相対 / 絶対 | **相対優先**、不可なら絶対 |
| 未知キー | **保持** |
| コメント | **保持非保証**（Phase 2 制限） |
| 未存在 DB path | **許可** |
| `db use --clear` | **採用** — default に復帰 |
| 親 dir 探索 | **Phase 2 以降・別 milestone** |
