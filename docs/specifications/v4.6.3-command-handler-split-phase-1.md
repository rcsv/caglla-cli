v4.6.2 で SQLite migration strategy を整理したあと、将来の **Tauri GUI 化**を見据えて CLI command handler と core / service logic の分離方針を整理する **documentation-first review** です。

| 実装 | **原則なし** — 大規模 handler 移動・GUI・schema 変更は行わない |

## Status

| 観点 | 判定 |
|---|---|
| 現行 handler 配置 | **`main.rs` 集中**（~1,078 行）— 大半の subcommand match |
| `commands/` 抽出 | **部分的** — `about`, `db` のみ |
| service layer | **未整備** — domain モジュールが repository + print + use case を兼ねる |
| GUI 再利用準備 | **不十分** — `println!` が handler / print_* に分散 |
| Phase 1 実装 | **defer** — 方針確定が先。必要なら read-only の最小 refactor のみ |

**結論:**

```text
CLI は残す。GUI は CLI の置き換えではなく、同じ service layer を呼ぶ別入口とする。
Phase 1 は地図作り。read-only command から service boundary を定義し、println を handler 末端へ寄せる。
write / import / delete / migration 系は最後。
```

## Background

### 完了済み hardening track

* v4.6.1 — FK / orphan data review
* v4.6.2 — migration strategy review

### なぜ handler split か

将来の入口候補:

```text
CLI
Tauri GUI
Web app
macOS app
test / fixture generator
```

`main.rs` の command handler に以下が混在すると、GUI から再利用しにくい:

```text
clap args parsing
DB access
domain validation
use case orchestration
terminal formatting (println)
JSON output mode selection
process exit behavior
```

v3.2.1 で `cli/args.rs`・`commands/db`・`output/json` 等への整理は済んだが、**handler 本体は main.rs に残存**（long-term strategy の v3.2.x defer 項目）。

---

## Goals

1. 現行 command handler 構造の棚卸し
2. CLI-only と reusable core/service の責務分類
3. Tauri GUI 初号機で再利用する use case の優先順位
4. **read-only first** の安全な split 順序
5. output model / error handling / test strategy の将来方針

**非目標:** Tauri 作成、GUI 実装、DB schema 変更、CLI breaking change。

## Current command handler structure

### エントリポイント

```text
main()
  → Cli::parse()                    [cli/args.rs]
  → resolve_db_path_for_cli
  → commands::db::run_before_open_db [db path/status/use のみ DB 前処理]
  → open_db_at
  → match Command { ... }           [main.rs — 大半]
```

### モジュール配置（v4.6.2 時点）

| 領域 | 場所 | 役割 |
|---|---|---|
| Clap 定義 | `src/cli/args.rs` | subcommand / flags |
| Handler（一部） | `src/commands/{about,db}.rs` | db path/status/use/reset |
| Handler（大半） | `src/main.rs` | trip〜participant の match arms |
| Repository + print | `src/{trip,itinerary,...}.rs` | SQL + `print_*` 混在 |
| Analysis | `src/analysis/{statistics,doctor,advisor}.rs` | 集計・診断（`run_*` は json 分岐あり） |
| IO | `src/io/{markdown,diff}.rs` | export-md / diff |
| JSON 出力 | `src/output/json.rs` | `print_json` |
| Domain types | `src/domain/models.rs` | エンティティ・export schema |

### 規模感

| ファイル | 行数（概算） | 備考 |
|---|---|---|
| `main.rs` | ~1,078 | 全 command match |
| `trip.rs` | ~3,229 | import/export/duplicate + print |
| `itinerary.rs` | ~2,990 | CRUD + timeline print |
| `domain/models.rs` | ~1,397 | 型定義集約 |

### 既存の良いパターン

| パターン | 例 |
|---|---|
| handler 薄化 + module `run_*` | `doctor::run_trip_doctor`, `advisor::run_trip_advisor` |
| JSON 分岐を handler で選択 | `trip list` — `json` flag → `print_json` vs `print_trip_list` |
| IO 分離 | `write_trip_export`, `write_trip_markdown` — ファイル/stdout |
| commands 部分抽出 | `db` — `run_before_open_db` / `run_after_open` |

### 混在パターン（改善対象）

| 問題 | 例 |
|---|---|
| handler 内 inline println | `itinerary add` — 10 行以上の成功表示 |
| domain module 内 print | `trip::print_trip_list`, `itinerary::print_itinerary_timeline` |
| service が terminal を知る | `statistics::print_trip_stats(conn, ...)` — DB + 表示 |
| analysis `run_*` が json + human 両方 | doctor/advisor — 許容だが GUI は report struct のみ欲しい |

## Responsibility boundaries

### 目標レイヤ

```text
CLI layer:
  clap args parsing
  human / JSON output mode
  stdout / stderr
  exit code 変換

Application / service layer:
  use case orchestration
  validation
  structured result 返却

Repository / DB layer:
  SQLite query / transaction
  row mapping

Domain layer:
  Trip / Day / Itinerary / Expense / Receipt 等
  domain rules / pure calculation
```

### 目標呼び出し

```text
CLI command  → service/usecase → repository → SQLite
Tauri command → same service/usecase → repository → SQLite
```

### 現状マッピング

| レイヤ | 現状の主な所在 |
|---|---|
| CLI | `cli/args.rs` + `main.rs` match + 成功 `println!` |
| Service | **明示層なし** — `trip.rs` import 等が use case 兼ねる |
| Repository | 各 `src/*.rs` の `add_` / `list_` / `get_` / `delete_` |
| Domain | `domain/models.rs` + 各 module の validation |

### `domain/models.rs` について

型定義が ~1,400 行に集約。v4.6.3 では **分割は defer**（long-term strategy 参照）。Phase 1 では service result 型を **新設する場合も domain サブモジュールまたは analysis 近傍** とし、models.rs 肥大化を避ける。

## Read-only first strategy

**Phase 1 実装（将来）でも read-only から着手する。**

理由:

* 副作用が少ない
* GUI 初号機に直結
* JSON 出力との親和性が高い
* integration test しやすい
* delete / import / migration より安全

最初の pilot 候補（推奨）:

```text
1. trip list / show
2. day list
3. itinerary list / timeline
4. trip stats
```

各 pilot で:

1. service 関数が **struct result** を返す
2. CLI handler は `json` → `print_json(result)` / human → `print_*`（既存文言維持）
3. **CLI 出力 golden / integration test 不変** を確認

---

## Output model strategy

### 原則

```text
CLI human output → CLI layer
CLI JSON output → serialization（output/json）
GUI → service result を UI model に変換
service layer は terminal を知らない
```

### 将来 result 型（v4.6.3 では型追加しない — 設計メモのみ）

```text
TripListServiceResult      { trips: Vec<Trip> }
TripDetailServiceResult    { trip: Trip }
DayListServiceResult       { trip_id, days: Vec<DayListEntry> }
ItineraryTimelineResult    { trip_name, items: Vec<ItineraryItem> }
TravelBookPreviewResult    { markdown: String }  // または structured sections
TripStatsServiceResult     { stats: TripStats }  // 既存 TripStats を再利用可
DoctorServiceResult        { report: DoctorReport }
```

### JSON 互換性

* 既存 `--json` の serde shape は **breaking 変更しない**
* service result は既存 domain type をラップする形が最小差分

---

## Error handling strategy

### 目標

```text
service layer:
  anyhow::Result または typed error で domain / validation / repository 失敗を返す

CLI layer:
  ? 伝播 → human-readable 表示（anyhow default）
  exit code は main の Result で統一（現状維持）

GUI layer:
  error を dialog / toast / inline message に変換
```

### Phase 1

* 既存 `anyhow::Result` パターンを **壊さない**
* service 導入時も `Result<T>` 返却で CLI handler の `?` は維持
* typed error は Tier 4 以降または必要になってから

## Test strategy

### 分離後に守るテスト層

| 層 | テスト種別 |
|---|---|
| Service | unit / integration — struct result の中身 |
| Repository | DB fixture — CRUD 正確性 |
| CLI | `tests/*_cli.rs` — stdout / exit code |
| Golden | export-md / export JSON — **Phase 1 で不変** |
| JSON | `--json` shape — serde roundtrip |

### Phase 1 ルール

```text
CLI 出力を変える refactor は golden 差分を生む → 避ける。
public command の breaking change 禁止。
tests/*_cli.rs の assert 文言は維持。
```

### 既存資産

* `tests/trip_stats_cli.rs`, `tests/doctor_advisor_json_cli.rs` 等 — JSON schema_version 固定
* `samples/okinawa_sesoko_2026/` — export golden
* `storage/db.rs` legacy_smoke — DB migration 系（handler split とは独立）

## Recommended split order

```text
1. command handler inventory（本書）
2. read-only command の service boundary 定義（docs + 型メモ）
3. Tier 1 pilot: trip list/show, day list, itinerary list/timeline, trip stats
4. CLI の println を handler 末端に集約（print_* は CLI adapter として残しても可）
5. service から println / eprintln 排除
6. Tier 2: doctor/advisor, export-md preview, receipt list
7. Tier 3: write commands
8. Tier 4: delete / import / duplicate / receipt assign
9. main.rs を commands/trip.rs 等へ **段階的** 移動（一括移動はしない）
```

### main.rs 分割方針（v3.2.x defer の更新）

```text
一括 commands/ 移動はしない。
read-only trip 系から commands/trip.rs（または handlers/trip.rs）へ pilot 抽出。
各段階で make check + CLI tests 全通過を gate とする。
```

## Out of scope for v4.6.3

* Tauri プロジェクト作成 / GUI 実装
* 大規模 handler 移動（main.rs 一括分割）
* service result 型の mass 導入
* DB schema / migration runner / FK
* CLI 出力仕様変更
* golden 大規模更新
* `domain/models.rs` 分割

### Optional minimal implementation（許容条件）

以下のみ、**public CLI behavior が完全不変** の場合に限り許容:

* pure helper の抽出
* println を伴わない計算の関数化
* read-only の private function 分割

該当しなければ **docs-only で完了**。
