v4.6.1 で FK / orphan data のリスクを棚卸ししたあと、SQLite DB を **安全に進化させるための migration strategy** を整理する **documentation-only review** です。

## Status

| 観点 | 判定 |
|---|---|
| 現行 migration 方式 | **暗黙の migrate-on-open** — `open_db_at` → `init_db` で毎回 idempotent 処理 |
| SQLite schema version 管理 | **未整備** — `PRAGMA user_version` / `schema_migrations` テーブルなし |
| `db status` の `schema_version` | **SQLite 版ではない** — CLI JSON 用定数（`DB_STATUS_JSON_SCHEMA_VERSION = 2`） |
| table rebuild 先例 | **あり** — `migrate_receipts_simplified_schema` |
| FK 導入の前提 | **migration strategy + orphan detection が先**（v4.6.1 結論） |
| 即 migration runner 実装 | **非推奨** — 方針確定と report-only orphan check が先 |

**結論（推奨）:**

```text
短期: PRAGMA user_version で DB schema version を管理し、明示的 migration step に移行する。
中期: backup / dry-run / preflight / report-only orphan detection を整えてから migration runner を最小実装する。
長期: 安全性が確認できた migration のみ auto-migrate を検討し、table-by-table FK を導入する。
```

---

## Background

v4.6.1 の結論:

* DB FK は部分的のみ。多くの子テーブルは FK 未整備。
* CLI の通常 delete はアプリ側 cascade で概ね安全。
* DB 直操作・将来 GUI / Web / 別プロセス利用では orphan risk が高い。
* **FK を即追加せず、migration strategy を先に設計する。**

v4.6.2 の主眼は、将来 FK / orphan hardening を安全に実装するために、**SQLite DB をどう進化させるかの地図**を作ること。CLI だけを前提にした暗黙の schema 更新は、GUI / Web / Tauri app が同じ DB を触る段階で危険になる。

## Current schema initialization

### エントリポイント

```text
open_db_at(path)
  → PRAGMA foreign_keys = ON
  → init_db(conn)
```

実装: `src/storage/db.rs`

### 初回作成と既存 DB オープン

**分離されていない。** 新規・既存とも `init_db` が同じ経路。

1. `CREATE TABLE IF NOT EXISTS` でコアテーブル作成（`trips`, `itinerary_items`, `checklist_items`, `days`, `notes`, `expenses`, `reservations`, `estimates`）
2. 続けて各 `migrate_*` を **毎回** 実行

### migrate-on-open の一覧（`init_db` 順）

| 順序 | 関数 | モジュール | 方式 |
|---|---|---|---|
| 1 | `migrate_itinerary_items` | `db.rs` | `ALTER TABLE ADD COLUMN`（idempotent） |
| 2 | `migrate_summaries` | `db.rs` | 列追加 / `RENAME COLUMN` |
| 3 | `migrate_days` | `day.rs` | 既存 Trip へ Day backfill |
| 4 | `migrate_itinerary_day_id` | `db.rs` | 列追加 + UPDATE backfill（未解決あれば bail） |
| 5 | `migrate_indexes` | `db.rs` | `CREATE INDEX IF NOT EXISTS` |
| 6 | `migrate_participants` | `participant.rs` | `CREATE TABLE IF NOT EXISTS` |
| 7 | `migrate_expenses_shared_expense` | `expense.rs` | 列追加 + `expense_beneficiaries` 作成 |
| 8 | `migrate_estimates` | `estimate.rs` | `CREATE TABLE IF NOT EXISTS` |
| 9 | `migrate_receipts` | `receipt.rs` | テーブル作成 + 列追加 + データ正規化 |
| 10 | `migrate_receipts_simplified_schema` | `receipt.rs` | **table rebuild**（legacy 列検出時のみ） |

### version 管理

| 種類 | 現状 |
|---|---|
| SQLite `PRAGMA user_version` | **未使用** |
| `schema_migrations` テーブル | **なし** |
| `db status --json` の `schema_version` | 定数 `2`（**DB ファイルの migration 版ではない**） |
| trip export JSON `schema_version` | v8 等 — **export/import 用。SQLite とは別** |

`print_db_status_human` は明示:

```text
Trip export schema version: N (trip export JSON; not SQLite migration version)
```

### `PRAGMA foreign_keys`

`open_db_at` で接続ごとに `ON`。別プロセスが OFF のまま接続すると DB CASCADE すら効かない（v4.6.1 参照）。

### test / fixture DB

| 用途 | 方式 |
|---|---|
| 単体テスト | `open_db_at(":memory:")` が多い |
| legacy smoke | `storage/db.rs` の `legacy_smoke` — 古い DDL のファイルを `open_db_at` で開く |
| golden / sample | `samples/okinawa_sesoko_2026/seed.sh` → CLI で Trip 構築 |
| 開発 reset | `caglla db reset` — `reset_db`（全 DELETE、開発用） |

## Current migration coverage

### 強み

* 各 `migrate_*` はおおむね **idempotent**（`IF NOT EXISTS`、列存在チェック、legacy 列の有無チェック）
* legacy DB 向け smoke test がある（`test_init_db_legacy_days_without_summary_column` 等）
* table rebuild の実績がある（receipts 単純化）
* `with_transaction` ヘルパーあり（`src/storage/db.rs`）

### 弱み

* **どの migration まで適用済みか DB に記録されない**
* 毎回全 migration を走査 — 将来 step が増えると起動コスト・複雑性が増加
* migration 失敗時の **部分適用状態** が version なしでは判別困難
* **backup / dry-run / explicit migrate コマンドなし**
* destructive 変更（`DROP TABLE`）がコードパスに存在するが、ユーザー確認なし
* orphan 検出なしのまま backfill / FK 前提処理が走る可能性

### Migration risks

| リスク | 説明 |
|---|---|
| 暗黙 migrate-on-open | ユーザーが migration 実行を認識しない。GUI 同時起動時に競合しうる |
| version 未記録 | 途中失敗 DB の再開時にどこまで進んだか不明 |
| DDL と transaction | SQLite は一部 DDL で transaction が commit される。複合 migration の原子性に注意 |
| backfill 失敗 | `migrate_itinerary_day_id` は未解決件数で **起動拒否**（良い先例） |
| export vs DB 混同 | `db status` の `schema_version` が SQLite 版と誤解されうる |
| FK 導入 blocker | orphan があると table rebuild + FK 追加が失敗または不整合固定化 |

## Schema version strategy

### 候補比較

| 方式 | SQLite らしさ | 実装単純さ | 履歴 | re-run 防止 | CLI/GUI 共有 | テスト |
|---|---|---|---|---|---|---|
| **A. `PRAGMA user_version`** | ◎ | ◎ | △（単一整数のみ） | ○（step 番号で制御） | ◎ | ◎ |
| **B. `schema_migrations` テーブル** | ○ | △ | ◎ | ◎ | ○ | ◎ |
| **C. 両方併用** | ○ | △ | ◎ | ◎ | ○ | ◎ |

### 推奨方針

```text
短期（次の実装フェーズ）:
  PRAGMA user_version で DB schema version を管理する。
  各 migration step に整数 version を割り当て、適用済みなら skip する。

中長期（runner が複雑化したら）:
  schema_migrations テーブルで適用履歴・checksum を残すことを検討する。
```

**現行コードとの接続:**

* `DB_STATUS_JSON_SCHEMA_VERSION` は **SQLite user_version とは別物**として維持するか、将来 `user_version` を読み取って報告するよう統合する。
* trip export `schema_version`（v8）は **変更しない** — export/import 境界は独立。

### 初期 version マッピング案（設計メモ）

既存の暗黙 migration を論理 step に束ねる案（実装は defer）:

```text
user_version 0  → 未初期化 / 極古 DB
user_version 1  → コアテーブル + itinerary 列拡張
user_version 2  → days + summaries + day_id backfill
...
user_version N  → 現行 HEAD（receipts simplified 含む）
```

正確な番号付けは migration runner 実装時に legacy fixture 行列で確定する。

## Migration execution model

### 推奨原則

```text
migration は原則として version step ごとに transaction で実行する。
既存データを破壊する migration は避ける。
破壊的変更が必要な場合は backup / dry-run / explicit confirmation を必須にする。
```

### 整理

| 観点 | 推奨 |
|---|---|
| 適用単位 | **1 version ずつ順番**（スキップ禁止） |
| idempotent | **各 step 内は idempotent**、version 記録で再実行は skip |
| 失敗時 | 途中 version で停止。DB は backup から復元 or 修復方針を文書化 |
| transaction | data migration は `BEGIN…COMMIT`。DDL は SQLite 制約を考慮し step 設計 |
| schema vs data | **分離を推奨** — 例: 列追加（schema）→ backfill（data）を別 version に |
| destructive | **原則禁止**。table rebuild は preflight + backup 必須 |
| backup なし | **開発 DB のみ許容**。ユーザーデータ DB では不可 |

### SQLite DDL の注意

* `ALTER TABLE ADD COLUMN` は比較的安全
* FK 追加・`ON DELETE` 変更・列制約強化は **table rebuild** が必要なことが多い
* `migrate_receipts_simplified_schema` は `CREATE → INSERT SELECT → DROP → RENAME` パターンの先例

## Backup strategy

### 推奨

| 項目 | 方針 |
|---|---|
| タイミング | **destructive / table rebuild migration の直前**（将来は任意 migration 前も検討） |
| ファイル名 | `{db_basename}.backup-YYYYMMDD-HHMMSS` 例: `caglla.db.backup-20260702-143000` |
| 保存場所 | **DB と同じディレクトリ**（デフォルト）。設定で別 path も将来検討 |
| 失敗時 | **migration を中止** |
| CLI 上書き | 同名 backup がある場合は suffix 連番または中止 |
| GUI | 自動 migration 前に backup 作成 + 結果表示を必須 |

### 現状

backup 自動作成は **未実装**。`db reset` は開発用全削除のみ。

## Dry-run and preflight checks

v4.6.2 では **実装しない**。将来の preflight 候補:

| チェック | 目的 |
|---|---|
| current / target schema version | 適用要否 |
| pending migrations 一覧 | ユーザー可視化 |
| orphan data 検出 | FK / rebuild 前提（v4.6.1 接続） |
| duplicate / invalid reference | import 残骸・手編集 DB |
| required columns 欠落 | legacy DB 判定 |
| incompatible old DB | サポート外 DDL |
| backup 作成可否 | ディスク容量・権限 |
| post-migration `PRAGMA foreign_key_check` | FK 導入後の検証 |

### 将来コマンド候補（確定ではない）

既存 CLI は `caglla db {path,status,use,reset}`。命名規則に合わせた候補:

```text
caglla db status          # 既存 — 将来 user_version / pending を追加
caglla db check           # 新規候補 — orphan / FK check / preflight
caglla db migrate --dry-run
caglla db migrate
```

`trip db *` ではなく **`db` サブコマンド配下** が自然（DB ファイル全体の操作のため）。

## Orphan detection relationship

v4.6.1 からの接続:

```text
FK を追加する前に orphan detection が必要。
orphan がある DB に FK を追加すると migration が失敗する。
最初は report-only。自動 repair は後回し。
```

### 検出対象（優先度順）

1. `expenses.itinerary_id` → 不在 itinerary
2. `estimates.itinerary_id` / `reservations.itinerary_id`
3. `expense_beneficiaries` → 不在 expense / participant
4. `notes` polymorphic owner → 不在 trip / day / itinerary
5. `receipts.trip_id` / `day_id`
6. `participants.trip_id`
7. `expenses.paid_by_participant_id`

### doctor / advisor との境界

| 手段 | 向き |
|---|---|
| `db check` / migrate preflight | **DB 整合性** — orphan、version mismatch |
| `trip doctor` | **ledger hygiene** — pending receipts 等（v4.5.1） |

orphan は **db check 側を優先**。doctor への統合は report-only フェーズ後に検討。

### Receipt / Expense 境界（維持）

* Receipt Inbox 残存は orphan ではなく **hygiene**（v4.5.0 / v4.5.1）
* Expense は Actual の正本 — migration strategy でもこの境界は変えない

## Table rebuild strategy

FK 追加や列制約変更時の将来パターン（実装 defer）:

```text
1. preflight: orphan check + backup
2. PRAGMA foreign_keys = OFF（必要時のみ。接続・step 設計に注意）
3. CREATE TABLE new_* (... FK ...)
4. INSERT INTO new_* SELECT ... FROM old_*（変換・欠損行の扱いを明示）
5. DROP TABLE old_*
6. ALTER TABLE new_* RENAME TO old_name
7. CREATE INDEX / trigger 再作成
8. PRAGMA foreign_key_check
9. user_version bump
10. PRAGMA foreign_keys = ON（接続既定に戻す）
```

### 先例: `migrate_receipts_simplified_schema`

* legacy 列 `itinerary_id` の有無で分岐
* transaction 内で rebuild
* データ変換（`ignored` → `trashed_at`）を含む

### ロールバック

SQLite migration は **原則ロールバック不可**。失敗時は **backup ファイルから復元** が現実的な復旧手段。

## CLI / GUI / Web migration boundary

### 推奨方針案

```text
初期段階:
  CLI の明示 migration command（caglla db migrate）を優先する。
  open 時の暗黙 migrate-on-open は段階的に廃止または read-only 検出へ移行する。
  GUI は古い DB を検出したら「migration が必要」と表示し、CLI 実行または明示操作を促す。

将来:
  安全性が確認できた migration のみ auto-migrate を検討する。
```

### 検討観点

| 観点 | 初期方針 |
|---|---|
| CLI 起動時 auto migrate | **現状維持だが deprecate 方向** — 明示 migrate へ |
| GUI 起動時 auto migrate | **しない** — 確認 UI + backup 後 |
| version mismatch | **read-only または拒否** — 黙って migrate しない |
| 複数プロセス | SQLite lock / `SQLITE_BUSY` — migration 中は単一 writer |
| migration 中 | **排他** — runner が file lock またはアプリ単一インスタンスを前提 |
| ユーザー確認 | destructive / major version bump で必須 |

### Tauri / Web

* Web はサーバー側 DB または sync 層になりうる — **同一 SQLite ファイル共有とは限らない**
* 共有する場合は **migration runner を一箇所に集約**（CLI か desktop shell）
