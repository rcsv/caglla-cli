## Status

| 観点 | 判定 |
|---|---|
| 現行 CLI の通常操作 | **概ね安全** — delete / import / duplicate はアプリ側 cascade を実装済み |
| DB schema の FK カバレッジ | **部分的** — `trips` 直下の一部のみ DB CASCADE |
| orphan 発生リスク | **中** — itinerary 配下・polymorphic notes・participants / receipts は FK なし |
| 別プロセスからの DB 操作 | **高リスク** — アプリ側防御だけでは不十分 |
| 即時 FK 追加 | **非推奨** — migration strategy と orphan 検出が先 |

**結論（推奨）:**

```text
現時点では FK 制約を即追加しない。
まず orphan risk を文書化し、migration strategy を先に設計する。
その後、doctor / validate / migration dry-run で orphan を検出できるようにする。
最後に、安全な順序で FK 制約や cascade policy を導入する。
```

---

## Background

Travel Ledger / Caglla.Travel は CLI 単体だけでなく、将来 **GUI / Web / macOS app** から同じ SQLite DB（または同等の永続化層）を触る可能性がある。

その段階では、command handler の delete 順序や repository の善意だけでは参照整合性を保証できない。DB 制約・migration 方針・診断手段を先に設計しておく必要がある。

v4.6.1 の主眼は「DB を強くすること」ではなく、**「DB を強くする前に、どこを触ると危ないかを把握すること」**。

## Current FK coverage

新規 DB（`CAGLLA_DB` で `db reset` 後）の `PRAGMA foreign_key_list` 結果:

| 子テーブル | 参照 | ON DELETE | DB に FK あり |
|---|---|---|---|
| `days` | `trip_id` → `trips.id` | **CASCADE** | はい |
| `itinerary_items` | `trip_id` → `trips.id` | **CASCADE** | はい |
| `itinerary_items` | `day_id` → `days.id` | **NO ACTION**（実質 RESTRICT） | はい（`migrate_itinerary_day_id` で列追加） |
| `checklist_items` | `trip_id` → `trips.id` | **CASCADE** | はい |
| `notes` | — | — | **いいえ** |
| `expenses` | — | — | **いいえ** |
| `expense_beneficiaries` | — | — | **いいえ** |
| `estimates` | — | — | **いいえ** |
| `reservations` | — | — | **いいえ** |
| `participants` | — | — | **いいえ** |
| `receipts` | — | — | **いいえ** |

### 補足

- `itinerary_items.day_id` は `INTEGER REFERENCES days(id)` として **後付け列**。`ON DELETE` は未指定のため **NO ACTION**。
- `notes` は polymorphic（`owner_type IN ('trip','day','itinerary')`）の CHECK のみ。コードコメントでも「FK は張らない」と明記（`src/note.rs`）。
- Receipt は Trip スコープの未整理証憑候補。Expense とは **assign で消費** する設計で、Receipt 行に `expense_id` FK は持たない（v3.7 単純化後）。

## Application-level cascade behavior

### `trip delete`（`src/trip.rs::delete_trip`）

トランザクション内の順序:

```text
1. delete_participants_for_trip
2. delete_notes_for_trip          （trip / day / itinerary 由来を subquery で削除）
3. delete_reservations_for_trip
4. delete_estimates_for_trip
5. delete_expenses_for_trip       （beneficiaries 含む）
6. delete_receipts_for_trip
7. DELETE FROM trips
   → DB CASCADE: days, itinerary_items, checklist_items
```

**アプリ側で先に消す必要があるもの:** notes, reservations, estimates, expenses (+ beneficiaries), receipts, participants。

**DB CASCADE に任せているもの:** days, itinerary_items, checklist_items。

`DELETE FROM trips` だけを実行すると、`expenses` / `reservations` / `estimates` / `notes` / `participants` / `receipts` が **orphan 化** する（FK なし）。

### `itinerary delete`（`src/itinerary.rs::delete_itinerary_item`）

```text
delete_notes_for_itinerary
delete_estimates_for_itinerary
delete_expenses_for_itinerary   （beneficiaries 含む）
delete_reservations_for_itinerary
DELETE FROM itinerary_items
```

### `day` 系（`src/day.rs::sync_days_to_trip_duration`）

旅行期間短縮時:

- Day に itinerary / title / summary がある場合は **エラーで拒否**
- 削除可能な Day について: `delete_notes_for_day` → `nullify_receipts_for_day`（`day_id` を NULL）→ `DELETE FROM days`
- `itinerary_items.day_id` FK により、参照中の Day は DB でも削除不可

### `expense delete` / `participant delete`

- `delete_expense`: beneficiaries を先に削除
- `delete_participant`: `paid_by` クリア + beneficiaries 削除後に participant 行削除

### `receipt delete` / `assign`

- `delete_receipt`: 単行削除
- `receipt assign`: Expense 作成後 **Receipt 削除**（消費モデル）
- `delete_receipts_for_trip`: trip delete から呼ばれる

### `db reset`（開発用、`reset_db`）

手動削除順序がドキュメント化されている:

```text
notes → reservations → expense_beneficiaries → expenses → estimates
→ participants → receipts → checklist_items → itinerary_items → days → trips
```

FK のない子を先に消し、最後に `trips` を削除する設計。

### import / duplicate

| 操作 | 整合性の扱い |
|---|---|
| `import_trip_from_export*` | 新 Trip 作成 → days → itinerary → 子エンティティの順で INSERT。バリデーションあり |
| `duplicate_trip` | export/import 相当の複製経路。ID は新規採番 |
| JSON import | schema 検証 + day_number / itinerary 範囲チェック |

### 整理: 誰が守っているか

| 整合性 | 現状 |
|---|---|
| Trip 削除時の itinerary 配下データ | **アプリ側**（必須） |
| Trip 削除時の days / checklist | **DB CASCADE** + アプリ順序 |
| Itinerary 削除時の expense 等 | **アプリ側** |
| Day 削除時の itinerary 参照 | **DB NO ACTION** + アプリ事前チェック |
| Note の polymorphic 参照 | **アプリ側**のみ |
| Participant ↔ Expense 参照 | **アプリ側**のみ |
| Receipt の trip / day 参照 | **アプリ側**のみ |

## Orphan data risk map

| シナリオ | 起き得るか | 深刻度 | 現状の防御 |
|---|---|---|---|
| Trip 削除後に Expense / Reservation / Estimate が残る | CLI 経由では低、raw SQL では **高** | 高 | `delete_trip` のアプリ cascade |
| Trip 削除後に Note が残る | 同上 | 中 | `delete_notes_for_trip` |
| Trip 削除後に Participant / Receipt が残る | 同上 | 中 | 明示削除 |
| Itinerary 削除後に Expense 等が残る | raw SQL では **高** | 高 | `delete_itinerary_item` |
| Day 削除後に Itinerary が浮く | **低** | 高 | FK NO ACTION + 期間短縮前チェック |
| Day 削除後に Receipt の `day_id` が dangling | **低**（意図的 NULL 化） | 低 | `nullify_receipts_for_day` |
| Note の `owner_id` が存在しない trip/day/itinerary を指す | import 不備 / raw SQL で **あり得る** | 中 | 削除時の明示処理、FK なし |
| `expense_beneficiaries` が存在しない expense / participant を指す | participant 削除以外で **あり得る** | 中 | participant 削除時の cleanup |
| `expenses.paid_by_participant_id` が削除済み participant を指す | **あり得る** | 低 | participant 削除時 `clear_paid_by` |
| Receipt が Trip と無関係な `trip_id` を持つ | raw SQL | 中 | FK なし |
| Receipt Inbox に未整理証憑が残り続ける | **仕様上あり得る** | 低（hygiene） | doctor/advisor（v4.5.1）、orphan ではない |
| import / duplicate の部分失敗 | **低** | 高 | トランザクション（要個別確認） |

**Receipt と Expense の関係:** assign 以外で Receipt が Expense に紐づくことはない。orphan というより **Inbox に残る未整理候補** は ledger hygiene の対象（v4.5.1 doctor/advisor）。

## SQLite-specific constraints

### 後から FK を追加しにくい

SQLite は既存テーブルに `ALTER TABLE ... ADD CONSTRAINT` で FK を後付けできない。一般的な手順:

```text
1. orphan 検出・修復
2. 新テーブル作成（FK 付き）
3. データコピー
4. 旧テーブル DROP
5. 新テーブル RENAME
```

既存データに orphan があると、FK 付きテーブルへのコピーや `PRAGMA foreign_key_check` で **migration が失敗** するか、不整合を固定化する。

### `PRAGMA foreign_keys = ON` は接続単位

- `open_db_at` では有効化済み
- 別プロセス（GUI、sqlite3 CLI、テストツール）が `foreign_keys` を OFF のまま接続すると、**DB CASCADE すら効かない**
- 将来の共有 DB 利用では、接続初期化の標準化が必要

### `itinerary_items` の二重参照

- `trip_id` + `day`（番号）+ `day_id`（行 ID）が併存
- `list_itinerary_items` は `days` と **INNER JOIN**（`day_id` 必須）
- `day_id` が NULL または dangling な itinerary は **一覧から見えなくなる** 可能性

### legacy DB

- 古い DB は `CREATE TABLE` 時点で FK が無いテーブルから始まり、migration で列追加
- `migrate_itinerary_day_id` は backfill 失敗時に **起動を拒否**（未解決件数で bail）
- ただし **expenses 等の orphan** は migration では検出しない

## Relationship with migration strategy

v4.6.2 候補「SQLite migration strategy review」との接続:

| v4.6.1 で確定すること | v4.6.2 以降に送ること |
|---|---|
| どの FK をいつ入れるかの優先順位 | `schema_version` 管理（DB ファイル内） |
| orphan が blocker であること | migration runner / backup before migrate |
| doctor / validate 連携の必要性 | dry-run migration check |
| テーブル単位の段階導入方針 | orphan detection query の標準化 |
| 別プロセス利用時の FK 必須化 | orphan repair policy（自動 vs 手動） |
| | migration test fixture / legacy DB 行列 |

**原則:**

```text
FK を入れるなら migration strategy が先に必要
orphan check を入れるなら diagnostic command または doctor 連携が必要
既存 DB を壊さない migration 設計が必要
```

現状、`db status` JSON の `schema_version`（`DB_STATUS_JSON_SCHEMA_VERSION = 2`）は **CLI ステータス表示用** であり、SQLite DDL のバージョン管理テーブルではない。v4.6.2 で整理が必要。

## Recommended hardening order

```text
1. v4.6.1: FK / orphan data hardening review（本書）          ← 今ここ
2. v4.6.2: SQLite migration strategy review
3. v4.6.x: orphan detection queries / doctor integration
4. v4.6.x: migration runner / backup / dry-run design
5. v4.7.x+: FK constraints introduction, table by table
```

### FK 導入の候補優先順位（提案）

| 順位 | 参照 | 推奨 ON DELETE | 理由 |
|---|---|---|---|
| 1 | `expenses.itinerary_id` → `itinerary_items` | CASCADE | orphan 影響大、削除パターン明確 |
| 2 | `estimates.itinerary_id` / `reservations.itinerary_id` | CASCADE | 同上 |
| 3 | `expense_beneficiaries` → `expenses`, `participants` | CASCADE | 従属データ |
| 4 | `participants.trip_id` → `trips` | CASCADE または RESTRICT+アプリ | trip 削除は既に明示削除 |
| 5 | `receipts.trip_id` → `trips` | CASCADE | trip 削除と整合 |
| 6 | `receipts.day_id` → `days` | **SET NULL** | 現行の `nullify_receipts_for_day` と一致 |
| 7 | `notes` polymorphic | **defer** | FK 化にはテーブル分割またはトリガが必要 |
| 8 | `itinerary_items.day_id` | SET NULL または RESTRICT | 現行は RESTRICT 相当で運用 |

**即追加しない理由:** 既存ユーザ DB に silent orphan がある可能性を先に検出・修復する手段がない。
