## Status

| 観点 | 判定 |
|---|---|
| service layer | **`src/services/trip_stats.rs`** 導入 |
| pilot command | `trip stats` が service 経由 |
| terminal I/O | service 内に `println!` / `eprintln!` なし |
| CLI / JSON 出力 | **不変** |

**結論:**

```text
trip stats は CLI → service → compute_trip_stats（repository 相当）の流れになった。
表示は CLI 層の print_trip_stats_display に閉じた。
他 read-only command は follow-up。
```

## Background

v4.6.3 で以下を確定した:

```text
CLI command  → service/usecase  → repository  → SQLite
Tauri command → same service/usecase → repository → SQLite
```

v4.6.4 は本格 split ではなく **pilot**。成功条件は「たくさん分離すること」ではなく、以下の確認:

- CLI behavior を壊さずに service boundary を導入できるか
- structured result を GUI に渡せる形にできるか
- `println` / formatting を CLI layer に閉じ込められるか

候補のうち **`trip stats`** を選んだ理由:

- v4.6.0 で `TripStats.days` semantics を修正済み
- `TripStats` が既に structured result として明確
- GUI dashboard / Travel Book 周辺で再利用価値が高い
- 集計と表示の分離効果が見えやすい

## Pilot target

**`trip stats`**（`TripAction::Stats`）

- human 出力: `trip stats <id>`
- JSON 出力: `trip stats <id> --json`

---

## Current flow before split

```text
main.rs (TripAction::Stats)
  ├─ --json → compute_trip_stats(conn, trip_id) → print_json(&TripStats)
  └─ human  → print_trip_stats(conn, trip_id)
                └─ compute_trip_stats + println!（DB + 表示が混在）
```

`compute_trip_stats` は既に structured result を返すが、CLI handler が直接呼び、human パスは `print_trip_stats` が DB 再取得相当の流れを内包していた。

## Service boundary introduced

### 新規モジュール

```text
src/services/mod.rs
src/services/trip_stats.rs
```

### API

```rust
pub struct TripStatsServiceResult {
    pub stats: TripStats,
}

pub fn get_trip_stats(conn: &Connection, trip_id: i64) -> Result<TripStatsServiceResult>
```

- `get_trip_stats` は `analysis::statistics::compute_trip_stats` を orchestrate
- service 内に terminal I/O なし
- 新型は最小 — 既存 `TripStats` を `stats` フィールドで包むのみ

### 表示分離

```text
analysis/statistics.rs:
  compute_trip_stats     — 集計（従来どおり、export-md 等も利用）
  print_trip_stats_display(stats) — human 出力のみ（println）
```

### After split

```text
main.rs (TripAction::Stats)
  → services::trip_stats::get_trip_stats(&conn, trip_id)
  ├─ --json → print_json(&result.stats)
  └─ human  → print_trip_stats_display(&result.stats)
```

## What changed

- `src/services/` モジュール追加
- `trip stats` handler が service 経由に
- `print_trip_stats(conn, …)` を廃止し `print_trip_stats_display(&TripStats)` に分離
- Cargo version `4.6.4`
- docs / release notes

---

## What did not change

- Tauri / GUI
- write command（delete / import / duplicate / receipt assign 等）
- DB schema / migration / FK
- `compute_trip_stats` の集計ロジック
- export-md の stats 取得（引き続き `compute_trip_stats` 直接）
- 他 read-only command（trip list / show 等）

---

## Lessons learned

1. **`compute_trip_stats` はほぼ service-ready だった** — pilot では thin wrapper で十分
2. **表示関数の `&TripStats` 化** が CLI/GUI 分離の最小ステップ
3. **`TripStatsServiceResult` は JSON 互換のため flatten せず `stats` フィールド** — CLI は `result.stats` を渡す
4. **export-md 等の非 pilot 呼び出しは触らない** — 変更範囲を pilot に限定できる
