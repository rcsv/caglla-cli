# Note list service boundary

v4.6.4〜v4.6.10 の read-only service boundary、および v4.6.11 review の推奨に従い、**`note list` 単独**で service 層を導入する。

| Item | Status |
|---|---|
| Target | v4.6.12 — note list service boundary |
| Target command | **`note list` のみ** |
| Scope | service 層導入（structured result）、CLI 出力互換維持 |

---

## Status

| 観点 | 判定 |
|---|---|
| service layer | **`src/services/note_list.rs`** 追加 |
| target command | `note list` が service 経由 |
| terminal I/O | service 内に `println!` / `eprintln!` なし |
| CLI human output | **不変** |
| CLI JSON output | **不変**（`NoteListJson` shape） |
| ordering | **不変**（`sort_order ASC, id ASC`） |
| owner resolution / not found | **不変** |
| DB schema / migration | **変更なし** |

---

## Background

目標形:

```text
CLI / Tauri
  → service (use case, structured result, no I/O)
  → repository / domain
```

`note list` は Trip / Day / Itinerary detail 画面に紐づくため、GUI 初号機で必要になる read-only 導線。v4.6.12 では Note の責務再設計は行わず、既存挙動を保ったまま service boundary のみ導入する。

---

## Target command

- `note list --trip <id>`
- `note list --trip <id> --day <n>`
- `note list --itinerary <id>`
- `--json` フラグあり（`NoteListJson`）

owner は `--trip` / `--trip + --day` / `--itinerary` のいずれか 1 つ。

---

## Current flow before split

```text
main.rs (NoteAction::List)
  → note::resolve_note_owner_for_list(conn, trip, day, itinerary)
  → note::list_notes_for_owner(conn, owner_type, owner_id)
  ├─ --json  → print_json(&NoteListJson { owner_type, owner_id, notes })
  └─ human   → note::print_note_list(owner_type, owner_id, &notes)
```

owner 解決・DB 取得・terminal 出力が handler に混在していた。

---

## Service boundary introduced

新規:

```text
src/services/note_list.rs
```

API:

```rust
pub struct NoteListServiceResult {
    pub owner_type: NoteOwnerType,
    pub owner_id: i64,
    pub notes: Vec<Note>,
}

pub fn list_notes(
    conn: &Connection,
    trip: Option<i64>,
    day: Option<i64>,
    itinerary: Option<i64>,
) -> Result<NoteListServiceResult>
```

service は owner 解決（`resolve_note_owner_for_list`）と一覧取得（`list_notes_for_owner`）を orchestrate し、**structured result のみ**返す。

`owner_type` / `owner_id` は human / JSON 両方で必要なため result に含める。JSON wrapper（`NoteListJson`）は CLI 層に残す。

---

## CLI compatibility

- human 出力: `note::print_note_list(result.owner_type, result.owner_id, &result.notes)`
- JSON 出力: `NoteListJson { owner_type, owner_id, notes }` を CLI 層で組み立てて serialize

いずれも従来と同一。

---

## JSON compatibility

- `--json` 時は `NoteListJson`（`owner_type`, `owner_id`, `notes`）を serialize（shape 不変）
- service result を wrapper 化して JSON に露出させない

---

## Error compatibility

- owner 未指定 / 不正: `resolve_note_owner_for_list` の既存メッセージ
  - 例: `owner を指定してください (--trip、--trip + --day、または --itinerary)`
- target 不存在: owner 解決内の `get_trip` / `find_day_by_id` / `get_itinerary_item` の既存メッセージ

---

## Ordering compatibility

`list_notes_for_owner` の SQL:

```text
ORDER BY sort_order ASC, id ASC
```

service は順序を変えない。

---

## Test coverage

- service test:
  - Trip / Day / Itinerary target で notes を返す
  - `sort_order ASC, id ASC` ordering 維持
  - empty target

## What changed

- `src/services/note_list.rs` 追加

---

## What did not change

- `note show` / `note add/update/delete`（対象外）
- `expense list` / `reservation list` / `checklist list` / `day show`（対象外）
- Travel Book export（対象外）
- write command / migration / FK（変更なし）
- Note 表示仕様・JSON shape（変更なし）

---

## Lessons learned

- owner 解決を service に移すと handler が薄くなる
- owner metadata（type / id）は human / JSON 双方で必要 → result に含める
- JSON wrapper（`NoteListJson`）は CLI 層に残し、互換性を守る
- `reservation list` と異なり `note list` は単一 target・単純 ordering で service 化が容易
