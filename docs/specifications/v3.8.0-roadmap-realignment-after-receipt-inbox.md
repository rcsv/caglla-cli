# v3.8.0 Roadmap Realignment after Receipt Inbox

Caglla.Travel CLI **v3.8.0** は **documentation-only release** です。v3.7.1 で Receipt Inbox workflow（assign / trash / pending sum / Okinawa sample）が一段落したあと、**製品メジャー版ロードマップ** と **次に着手してよい設計テーマ** を再整列します。

|---|---|

| **本書** | v3.7.1 後のロードマップ矛盾の整理・候補テーマの開始可否 |

設計系列:

```text
v3.7.1 Receipt Inbox 完了     → tag v3.7.1
```

## 1. Summary

v3.7 系列で Receipt Inbox は **metadata-only（v3.6.0）→ assign / trash workflow（v3.7.0）→ canonical sample（v3.7.1）** まで到達した。次の大きな機能実装に入る前に、長期ロードマップの **v4 Reservation 矛盾** を解消し、v4 以降を **未実装の到達点** だけが並ぶ状態に戻す。

**結論:**

| 項目 | 判定 |
|---|---|
| v4 = Reservation は **廃止**（v1.18.0 で実装済み） | ✅ |
| v4 = **Travel Book** に差し替え | ✅ |
| v5–v8 を 1 段繰り上げ（旧 v5–v9） | ✅ |
| v3.8.0 は **コード変更なし** | ✅ |
| 次候補は **設計着手可** と **着手不可** を分離 | ✅ |

## 3. Problem: v4 Reservation contradiction

[long-term-version-strategy.md](../long-term-version-strategy.md)（v3.7.1 時点）では:

```text
v4 — Reservation（予約情報の正式管理）
  Reservation Entity / Export / Display
```

一方、実装は **v1.18.0** で既に存在する:

```text
reservation add/list/show/update/delete
Itinerary 配下保存、Trip 一覧集約
export/import roundtrip
```

### 矛盾の原因

本ロードマップの **製品メジャー番号（v4, v5, …）** と **CLI パッチバージョン（v1.18.0 等）** は別軸である旨は注記されていたが、v4 が Reservation のまま残り、**「未実装の次メジャー」** と誤読されやすかった。

### 解消方針

1. **v4 Reservation をロードマップから退役** — 「v1.18.0 で到達済み」と明記
2. **旧 v5 Travel Book を v4 に繰り上げ**
3. **旧 v6–v9 を v5–v8 に繰り上げ**
4. 早見表と [current-work.md](../current-work.md) の **Next / Do not start** を同期

## 4. Realigned forward roadmap (v4+)

製品メジャー版の **これから** の到達イメージ（実装スケジュールの確約ではない）:

| メジャー | テーマ | ユーザーができること（イメージ） | 旧番号 |
|---|---|---|---|
| **v4** | **Travel Book** | 旅行 **前** の共有しおり（Rich MD / PDF、Summary・Reservation の統合表示） | 旧 v5 |
| **v5** | **Travel Journal** | 旅行 **後** の記録（Photo / Attachment、体験・感想） | 旧 v6 |
| **v6** | **Identity** | 利用者・アカウント（クラウド同期はまだしない） | 旧 v7 |
| **v7** | **Cloud** | 同期・バックアップ・共有 | 旧 v8 |
| **v8** | **Platform** | Desktop / Mobile / API / Caglla Engine | 旧 v9 |

### v4 Travel Book — 位置づけ

[summary-post-implementation-review.md](summary-post-implementation-review.md) の整理と整合:

```text
Summary     = Abstract（要旨・スキャン用）
Note        = Annotation
export-md   = Travel Book Generator v0（現状）
Travel Book = 共有用しおり（v4 到達点 — 出力物の強化）
```

現行 `trip export-md` は **Travel Book の原型** として扱う。v4 では PDF・レイアウト・Reservation / Summary のしおり向け統合など **出力パイプライン** を強化する想定。**新エンティティ必須ではない** 場合が多い。

### v5 Travel Journal — ブロッカー

Photo / Attachment は **Evidence 共通レイヤー** の設計が先（[v3.5.0-receipt-inbox-concept-design.md](v3.5.0-receipt-inbox-concept-design.md) §7、[v3.6.0-receipt-inbox-metadata-only-post-implementation-review.md](v3.6.0-receipt-inbox-metadata-only-post-implementation-review.md)）。Receipt 専用 `image_path` 先行実装は **採用しない**。

## 5. v3.7.x 後の候補テーマ（開始可否）

v3.8.0 時点での **優先検討リスト**。実装着手は別リリースで判断する。

### 5.1 評価表

| 候補 | 種別 | 設計着手 | 実装着手 | 優先度 | 備考 |
|---|---|---|---|---|---|
| **DB path 切替** | v3.x 基盤 | **可** | 未 | **高** | `--db` / `CAGLLA_DB` / `db use` — v3.2.0 から defer。`db path` / `db status` と自然に接続 |
| **doctor / advisor 活用** | 診断強化 | **可** | 未 | **中** | Estimate / Receipt / Pending の **助言** のみ。stats Actual への Receipt 混入はしない |
| **Travel Book（v4）** | 製品メジャー | **可** | v4 スコープ | **中** | export-md 拡張・PDF 等の **概念設計** から。Reservation は既存 CRUD を統合表示 |
| **Day 単位 Planned vs Actual** | 表示 | 後回し | 未 | 低 | Trip / Itinerary 差分は v3.3 / v3.4 で完了 |
| **receipt purge** | Receipt | 未 | 未 | 低 | trash で十分な間は不要 |
| **Travel Journal（v5）** | 製品メジャー | **不可** | **不可** | — | Evidence / Attachment 設計が先 |
| **Evidence / Attachment / image_path** | 基盤 | **不可** | **不可** | — | Receipt / Expense 共通レイヤーとして将来設計 |
| **OCR / automatic receipt parsing** | 自動化 | **不可** | **不可** | — | Evidence より後 |
| **Balance / Settlement** | 精算 | **不可** | **不可** | — | Shared Expense v3.0.0 以降も defer 継続 |
| **Expense reassign / unassign / trash** | Expense | **不可** | **不可** | — | Receipt assign とは別問題 |
| **trip stats への Receipt 反映** | 集計 | **不可** | **不可** | — | Pending は Actual ではない |

### 5.2 推奨する次の設計トピック（いずれか 1 つから）

1. **DB path 切替** — スコープが明確、v3.2 系の自然な延長、canonical sample / CI との混同防止に直結
2. **Travel Book v4 concept design** — ロードマップ v4 の正式な設計書（export-md を Generator v0 と位置づけ）
3. **doctor-advisor Receipt / Estimate utilization design** — 診断メッセージとコマンドヒントの拡張のみ（集計変更なし）

**同時並行は 1 テーマ推奨。** v3.8.0 リリース後は [current-work.md](../current-work.md) の Next action で 1 つを選ぶ。

### 5.3 DB path 切替（概要）

v3.2.0 で `db path` / `db status` を追加済み。未実装の延長:

```text
--db <path>     グローバルまたはサブコマンド共通フラグ（要設計）
CAGLLA_DB       環境変数（要設計）
db use <path>   永続設定（要設計 — 設定ファイルの有無も含む）
```

**開始可否:** 概念設計・実装計画の作成は **可**。`db reset` や open 前 dispatch との整合を先に文書化する。

### 5.4 doctor / advisor 活用（概要）

現行 `trip doctor` / `trip advisor` は Itinerary 完全性・restaurant 等が中心。defer されてきた拡張:

```text
未整理 Receipt が多い Trip への INFO / 助言
Estimate あり・Expense なし Itinerary への軽い注意
Pending Receipt sum の表示（stats ではなく advisor 助言のみ）
```

**開始可否:** 設計書の作成は **可**。**trip stats / export-md の Actual・Difference 定義は変更しない**。

### 5.5 Travel Book（概要）

```text
現状: trip export-md = しおり MD（Summary・Reservation・Planned/Actual 含む）
v4:   レイアウト・PDF・共有向けテンプレート・「旅行前ドキュメント」としての完成度
```

**開始可否:** v4 concept design の作成は **可**。実装は v4 リリース計画で切る。

## 6. Canonical “Do not start yet”

```text

OCR
automatic receipt parsing
Balance / Settlement（精算・振込計算）
Participant sharing 拡張（Settlement 連動）
Expense reassign / unassign / trash
receipt purge
Travel Journal 実装（v5 — Evidence / Attachment 未設計）
trip stats / Planned vs Actual への Receipt・Pending 反映
Potential Actual 表示
Cloud / Identity / Platform 実装
```

**設計着手は可・実装は不可**（上記に含めない）:

```text
DB path 切替（設計・計画）
doctor / advisor Estimate・Receipt 活用（設計・計画）
Travel Book v4 concept design（設計・計画）
```

## 7. v3.7 Receipt Inbox — closed scope

以下は v3.7 系列で **完了** とみなし、v3.8+ のデフォルト候補に **戻さない**:

```text

receipt assign / trash / restore
pending sum（receipt list）

```
