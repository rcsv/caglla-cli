## Background

v4.6.3 で `main.rs` の command handler split Phase 1 を完了したあと、v4.6.4 から read-only command を 1 コマンドずつ service 層へ切り出してきた。

目標形:

```text
CLI / Tauri
  → service (use case, structured result, no I/O)
  → repository / domain (DB query, ordering, validation)
```

v4.6.11 は追加実装に進む前に、7 モジュールの棚卸しと責務境界の確認を行う。

## CLI / service / repository boundary

### 責務分担（確立済みパターン）

```text
CLI layer (main.rs):
  - clap args 解釈
  - --json / human 選択
  - print_json(&existing_type)
  - print_*_display / print_*_detail / print_*_list
  - human-only 追加セクション（Reservations 等）
  - 見出し println!（例: 「{trip.name} のタイムライン:」）

service layer (src/services/):
  - use case orchestration
  - structured ServiceResult
  - repository / domain 関数呼び出し
  - no terminal I/O

repository / domain layer:
  - DB query / mutation
  - domain model
  - ordering（例: ITINERARY_LIST_ORDER_BY）
  - validation / not found error message
```

### 意図的に CLI に残した判断

| 判断 | 理由 |
|---|---|
| **itinerary show の Reservations** | JSON は `ItineraryItem` 単体。予約一覧は human-only 表示。service に含めると JSON shape リスク |
| **day list の DayListJson 組み立て** | JSON wrapper（`trip_id`, `trip_name`, `days[]`）は CLI 専用 shape。service は `Trip` + `Vec<Day>` を返し、`day::print_day_list_json` が組み立て |
| **timeline の --json 非新設** | 従来から human only。GUI 化時に service result をそのまま Tauri へ渡す想定 |
| **list/show の JSON wrapper 化しない** | `trip list` → `Vec<Trip>`、`itinerary list` → `Vec<ItineraryItem>` 等、既存 shape を serialize。互換性最優先 |

### main.rs の service 呼び出し箇所（7 箇所）

```text
trip list      → services::trip_list::list_trips
trip show      → services::trip_show::show_trip
trip stats     → services::trip_stats::get_trip_stats
day list       → services::day_list::list_days
itinerary list → services::itinerary_list::list_itineraries
itinerary timeline → services::itinerary_timeline::get_timeline
itinerary show → services::itinerary_show::show_itinerary
```

## main.rs residual responsibilities

### 規模

```text
wc -l src/main.rs  →  1083 行（2026-07-02）
```

v4.6.3 Phase 1 以降も依然として command dispatch + output selection の中心。

### 残責務の分類

| 分類 | 例 | service 化 |
|---|---|---|
| **service 経由 read-only** | trip/day/itinerary list/show/stats/timeline | 済（7 コマンド） |
| **repository 直結 read-only** | note/expense/estimate/reservation/checklist list/show、day show | 未 |
| **write command** | add/update/delete/move/replicate 等 | 意図的に未（Tier 3+） |
| **分析・診断** | trip doctor, trip advisor | 未 |
| **I/O・import/export** | trip export/import/validate/diff, export-md | 未 |
| **terminal 出力** | 成功メッセージ、human display、JSON wrapper 組み立て | CLI 層として妥当 |

### 次に切るなら

1. **`note list`** — owner 解決 + list + `NoteListJson` が CLI に混在。read-only で軽い
2. **`day show`** — Day detail 画面向け。`DayShowJson` は CLI 専用 wrapper
3. **`checklist list`** — Trip detail タブ向け
4. **`reservation list` / `expense list`** — やや複雑（target 解決、context rows）

### write command に進む前の設計メモ

```text
- write service は validation + mutation + structured result（変更後 entity）が必要
- dry-run パターン（checklist generate 等）の service 化方針を先に決める
- transaction 境界（replicate, swap day 等）を service でどう表現するか
- read-only expansion を一通り終えてから write に進むのが安全
```

### 推奨: v4.6.12 — note list service boundary

**理由:**

```text
- Note は Trip / Day / Itinerary に紐づき、GUI detail 画面で必須になりやすい
- Expense / Reservation より表示責務が軽い（list + show、金額計算なし）
- owner 解決（trip/day/itinerary フラグ）は service に入れられる
- NoteListJson wrapper は CLI 層に残し、service は Vec<Note> + owner metadata を返す形が自然
- write command（note add/update/delete）には触れない
```

### Note / Expense / Reservation へ進む前の注意点

| 領域 | 注意点 |
|---|---|
| **Note** | `resolve_note_owner_for_list` の owner 解決を service に移す。`NoteListJson` は CLI |
| **Expense** | `ExpenseListJson`、target 解決、通貨表示。human list は formatter 依存 |
| **Reservation** | trip list は context rows 付き。itinerary list は単純。2 パスに分かれる |
| **共通** | JSON wrapper を service result に入れない。既存 shape を CLI serialize で維持 |
| **共通** | human-only 追加表示（itinerary show の Reservations パターン）を増やさない |

## Lessons learned

1. **薄い service + CLI 側 JSON/human 分岐** が互換性維持に有効
2. **ServiceResult は最小限** — wrapper 型を service に入れない
3. **human-only セクション**（Reservations）は CLI に残す判断は正しかった
4. **1 コマンド 1 リリース** で rollback と review が容易
5. **ordering は repository 層** — service は加工しない（itinerary 系で実証済み）
