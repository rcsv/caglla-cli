### 2.2 評価

- **十分に軽くなったか:** はい。表示 **判定**（visibility / dedup / column / ordering）は presentation 側に移動済み。`markdown.rs` は「文字列をどう組むか」に集中している。
- **残りすぎていないか:** 大きな残留はない。細かい **ラベル構成**（`format_note_heading` の `Trip —` / `Day N —`、`format_day_heading` のラベル、`format_trip_date_range`）は presentation 寄りだが、構文（`###`）と密結合しているため無理に剥がすと可読性が下がる。
- **helper 化しすぎか:** いいえ。呼び出し側は `if show_note_column { ... }` のように読みやすい。過剰分割の兆候なし。

### 2.3 分類

| 対象 | 分類 |
|---|---|
| 章組み立て（`append_*` / `format_*_chapter`） | **Keep in markdown.rs** |
| `format_note_heading` のラベル選択、`format_day_heading` ラベル、`format_trip_date_range` | **Move later** |
| pipe table / `####` / `- ` / `**bold**` / `- [ ]` / 空行 | **Do not move** |

## 3. `travel_book_presentation.rs` responsibility review

### 3.1 現状

| グループ | helper |
|---|---|
| Overview | `trip_overview_worth_showing`, `trip_overview_time_metrics_worth_showing` |
| Days overview | `collect_days_overview_entries`(+`DaysOverviewEntry`), `travel_book_day_overview_label` |
| Planned cost | `planned_cost_note_column_visible`, `planned_cost_estimate_display_title`, `planned_cost_itinerary_group_label` |
| Itinerary | `format_travel_book_category_detail_line` |
| Notes | `travel_book_note_sort_key`, `sort_export_notes_for_travel_book` |
| Reservation | `reservation_provider_line_redundant`, `format_travel_book_reservation_period`, `format_travel_book_reservation_heading`（+ private datetime parse） |

### 3.2 評価

- **集約できているか:** はい。renderer 非依存の表示判断が 1 モジュールに集まっている。
- **モジュール名と責務:** 一致。`travel_book_presentation` は presentation rule 置き場として妥当。
- **雑多な置き場になり始めているか:** 軽い兆候あり。13 helper がフラットに並び、グループ境界がコメントのみ。まだ許容範囲だが Phase 3 で増えると見通しが落ちる。
- **struct 化が必要か:** 現時点では **不要**。ただし 2 点で境界が見え始めている:
  1. `format_travel_book_category_detail_line` は文字列 `- 種別: X` を返しており、**bullet 構文（`- `）を含む**＝わずかに renderer 混入。純粋な presentation なら「ラベル + 値」を返し、`- ` は renderer が付けるべき。
  2. `format_travel_book_reservation_heading` は `**bold**` を含む＝同様の混入。
- **関数群で十分か:** Phase 3 までは十分。GUI/Web 再利用を本気で狙う段になったら struct 化が必要。

### 3.3 分類

| 対象 | 分類 |
|---|---|
| Overview / Days overview / Planned cost / Notes 判定群 | **Healthy** |
| モジュール内のグループ分け（サブモジュール or コメント整理） | **Needs cleanup**（軽微） |
| category / reservation heading の構文混入除去 | **Needs cleanup**（Phase 3 候補） |
| `TravelBookDocument` / `TravelBookItem` 型 | **Candidate for struct/model**（今は Defer） |

## 6. `TravelBookDocument` introduction timing

### 観点評価

| 観点 | 現状 |
|---|---|
| helper 関数群だけで限界か | **まだ限界ではない**。visibility / formatting / ordering は関数で十分表現できている |
| renderer に章組み立て責務が残りすぎか | **ある程度残る**が、これは Markdown renderer の正当な責務。GUI が現れて初めて「共通の章モデル」が要る |
| GUI/Web 再利用に struct が必要か | **将来 Yes**。ただし現時点で GUI 実装予定はなく、モデルの形を決める入力（UI 要件）がない |
| いま導入すると premature か | **Yes、premature**。UI 要件なしに `TravelBookItem` を作ると、Markdown 都合の形に固まるリスク |

### 結論

**Do not introduce yet** — ただし v4.4.6 で **もう一段の small extraction**（§5 R1–R4）を行い、`markdown.rs` の残留 presentation を出し切ってから、v4.5.0 で prototype 判断するのが安全。

推奨シーケンス:

```text
v4.4.6 — Phase 3 small helper extraction（note heading / day label / date range）
       → その後 markdown.rs が純粋 renderer にどこまで近づくか再評価
v4.5.0 — TravelBookDocument prototype は「UI 要件 or Venue 着手」が見えてから
```

   note heading / day label / date range の小さな抽出。golden 不変。
