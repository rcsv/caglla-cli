## 4. Okinawa canonical sample review

### 4.1 ストーリー意図

旅行中に土産物屋・売店のレシートが発生するが、細かく整理する余裕がない。まず Receipt Inbox に **未整理の支払い候補** として登録し、帰宅後に見返して整理する。

```text
Receipt = 未整理の支払い候補
Receipt のままでは Actual ではない
Pending Receipt sum も Actual ではない
assign 後に作成された Expense だけが Actual に入る
```

sample README に上記を明記。v3.7.0 workflow design の用語と一致。✅

### 4.2 seed 投入データ

| Receipt（seed 後） | Day | 金額 | 整理イメージ | 判定 |
|---|---|---|---|---|
| 美ら海水族館ショップ | 2 | ¥4,860 | assign → Itinerary `水族館に入館`（ID 19） | ✅ |
| ハナサキマルシェ | 2 | ¥2,340 | assign → Itinerary `スタバる`（ID 26） | ✅ |
| 道の駅 許田 | — | ¥3,180 | 立ち寄り Itinerary を後で判断 | ✅ |
| 那覇空港売店 | 4 | ¥5,600 | 旅行費用に含めるか迷う候補 | ✅ |
| コンビニのレシート | 4 | 金額未入力 | 金額確認後に assign | ✅ |
| 個人的な雑貨購入 | 3 | ¥1,200 | trash 済み（旅行共通費にしない） | ✅ |

- **active Receipt（金額あり）合計:** ¥15,980 — `receipt list --trip 1` の Pending summary と一致 ✅

### 5.2 trashed `trashed_at` export fix

**問題（v3.7.0）:** DB は `now_string()` 形式（`YYYY-MM-DD HH:MM:SS`）で `trashed_at` を保存するが、export はそのまま出力していた。`trip validate-export` は schema v8 で `trashed_at` を **RFC3339** 必須とするため、trashed Receipt を含む export が **無効** になり得た。

**修正（v3.7.1）:** `build_export_receipt_v7` が export 時に `format_db_timestamp_for_export` を適用。

| 入力 | 出力 |
|---|---|
| 既に RFC3339 | そのまま pass-through |
| SQLite naive datetime | ローカルタイムゾーン解釈 → RFC3339 |
| その他 | フォールバック（元文字列） |

| 観点 | 判定 |
|---|---|
| DB 内部表現は変更しない | ✅ |
| import 側は RFC3339 / DB 形式の両方を受け付け可能 | ✅ |
| export-only 変換（validate-export 互換） | ✅ |
| `created_at` / `updated_at` は対象外（従来どおり） | ✅ |

実装: [src/receipt.rs](../../src/receipt.rs) — `format_db_timestamp_for_export`、`build_export_receipt_v7`。
