## 6. TravelBookDocument timing decision

### 観点評価（v4.4.5 更新）

| 観点 | 現状（Phase 3 後） |
|---|---|
| helper 関数群の限界 | **まだ限界ではない**が、**実用的な抽出はほぼ完了**。残りは cleanup 2 件 + planned cost グルーピング程度 |
| Markdown renderer だけでは struct 必要性 | **弱い**。Markdown は章文字列の組み立てで十分機能している |
| GUI / Web / macOS app 要件 | **未着手**。UI 章レイアウト・インタラクション要件がない |
| Venue / map / schedule role | **未着手**。モデル形状の入力がない |
| 今始めると Markdown 都合の model になるリスク | **依然として高い** |
| helper 群が model 化を求めているか | **部分的**。planned cost グルーピングと reservation/category 構文分離は struct の芽だが、**単独では prototype 着手の十分条件にならない** |

### 結論

```text
Defer v4.5.0 TravelBookDocument prototype until UI / Venue requirements appear
```

Phase 1〜3 で helper extraction は **十分完了**。`TravelBookDocument` は **要件ドリブンで再開**すべき。Markdown 都合で先に struct を作るメリットは現時点で小さい。

## 8. Next milestone recommendation

**推奨: B（optional）→ Defer A**

```text
B. v4.4.8 — Travel Book presentation helper cleanup（推奨・次）
   既知 2 件の Markdown 構文混入を是正（category `-`、reservation `**`）。
   golden 不変の small cleanup。v4.4.x helper extraction arc をきれいに閉じる。

A. v4.5.0 — TravelBookDocument prototype
   → Defer。UI / Venue / macOS native app 要件が見えてから再判断。

C. Defer TravelBookDocument and return to another roadmap item
   → v4.4.8 をスキップし、Travel Book presentation を一旦凍結して別ロードマップへ。
   妥当な選択肢だが、2 件の既知 cleanup は小さく残る。
```

**理由:**

1. Phase 1〜3 の helper extraction は **目的を達成** — `markdown.rs` は renderer として十分軽い
2. v4.4.5 から指摘の 2 構文漏れのみが **未処理の既知課題** — v4.4.8 で閉じられる
3. `TravelBookDocument` は UI 要件なしでは **依然 premature** — v4.5.0 着手は推奨しない
4. planned cost グルーピングの struct 化は **prototype 着手時**にまとめて扱う方が自然

**v4.4.x シリーズの締め:** v4.4.8 cleanup（任意）→ Travel Book presentation helper extraction **完了宣言** → v4.5.0 は要件待ち。
