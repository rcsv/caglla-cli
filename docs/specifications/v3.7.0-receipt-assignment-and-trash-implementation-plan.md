## 1. Background

- v3.6.0 で Receipt Inbox metadata-only support を実装した
- v3.6.0 の Receipt は **Trip-level collection**（Itinerary 配下にネストしない）
- Receipt は **Actual ではない**
- Receipt は Planned vs Actual / `trip stats` / `trip export-md` に含めない（差分計算は Estimate + Expense）
- v3.7.0 workflow design で、Receipt Inbox の次の操作として **assign / trash / pending sum** を整理した
- `ignored` status は、trash を導入すると役割が重複しやすく **将来的に廃止候補**になった

---

## 2. Goal

v3.7.0 の目的:

```text
Receipt Inbox に残っている未整理支払い候補を、
Expense 化するか、Trash に移動するか、未整理として残すか、
自然に扱える workflow を実装する。
```

主な goal:

- `receipt assign` を設計する（Receipt → Expense 作成 + Receipt 削除）
- `receipt trash` を設計する（Inbox から除外するが物理削除はしない）
- `receipt restore` / `receipt purge` を検討する（Trash 導入の整合）
- Pending Receipt sum（active Receipt 合計）を設計する（Actual ではない補助情報）
- v3.6.0 `ignored` status の移行方針（`ignored -> trashed`）を設計する
- Receipt は引き続き Actual に含めない（Expense 化された後だけ Actual に反映）

## 4. Receipt lifecycle target

v3.7.0 以降の Receipt lifecycle（推奨）:

```text
active Receipt:
  Inbox に残る未整理支払い候補（pending）

trashed Receipt:
  不要としてゴミ箱に移動された Receipt

assigned Receipt:
  Receipt レコードとしては残らない
  Expense が作成され、Receipt は削除される
```

DB 案:

```text
receipts
  ...
  trashed_at optional
```

意味:

```text
trashed_at IS NULL     -> active Inbox Receipt
trashed_at IS NOT NULL -> Trash
```

`status` は将来的に削除または段階的に縮小する方針とする。

v3.6.0 互換として現存する `ignored` は、migration で `trashed_at` へ移行する候補（推奨: `ignored -> trashed`）。

---

## 5. Migration plan

### 現状（v3.6.0）

```text
status = unreviewed | ignored
```

### v3.7.0 候補

```text
active/trash 判定を trashed_at に寄せる
status は縮小/廃止（段階的）
trashed_at を追加する
```

推奨 migration:

```text
unreviewed -> trashed_at NULL
ignored    -> trashed_at = migration timestamp
```

理由:

- v3.6.0 の ignored は「対象外として処理済み」
- v3.7.0 では対象外 Receipt は Trash に相当する
- したがって `ignored -> trashed` が自然

論点:

- export/import schema の更新が必要になる可能性（`trashed_at` を export に持つか）
- `status` を export に残すか、移行後に drop するか（段階的縮小の進め方）

## 6. `receipt assign` plan

`receipt assign` は Receipt を Itinerary に link する操作ではない。

```text
Receipt から Expense を作成し、Receipt を Inbox から消す（削除する）操作
```

CLI 候補:

```bash
caglla receipt assign <receipt_id> --itinerary <itinerary_id>
```

補完候補（Receipt に不足がある場合）:

```bash
caglla receipt assign <receipt_id> --itinerary <itinerary_id> --amount 1700 --currency JPY --memo "おかんのお土産"
```

設計方針:

- assign 先は Itinerary（`--itinerary` 必須を優先）
- Receipt の `amount` / `currency` / `memo` / `occurred_date` を Expense に引き継ぐ
- Receipt に `amount` / `currency` がない場合は assign 時に補えるようにする（ただしペア必須）
- Expense 側の必須項目が不足する場合は assign を失敗させる
- assign 後、Receipt は削除する
- `linked` / `converted` status は作らない
- `linked_expense_id` は復活させない

---

## 7. Transaction plan

`receipt assign` は必ず transaction。

処理イメージ:

```text
BEGIN
  create Expense from Receipt
  delete Receipt
COMMIT
```

失敗時:

```text
ROLLBACK
```

避けるべき状態:

```text
Expense は作成されたが Receipt が残る
Receipt は削除されたが Expense 作成に失敗する
```

---

## 8. Field mapping

Receipt → Expense の mapping 案（既存 Expense のフィールド命名に合わせて最終確定する）:

| Receipt field | Expense mapping | 方針 |
|---|---|---|
| `amount` | `Expense.amount` | 引き継ぐ（不足時は assign で補完） |
| `currency` | `Expense.currency` | 引き継ぐ（不足時は assign で補完） |
| `memo` | `Expense.note` または `Expense.title/note` | 既存 CLI の semantics に合わせる（title 必須なら assign で補完が必要） |
| `occurred_date` | Expense 側に日付概念があれば利用 | 無ければ Itinerary の文脈（日付）に吸収し、note に残す |
| `day_id` | Itinerary 候補の絞り込みに利用 | Expense へは直接引き継がない |
| `created_at` / `updated_at` | 引き継がない | Expense は新規作成時刻を持つ |

**要確認ポイント（実装前に確定）**:

- Expense 作成に必要な最小入力（title / category / date 等）
- Receipt の情報だけで足りない場合に、assign でどのフラグを要求するか

## 9. Day / Itinerary selection

Receipt の `day_id` は Itinerary 選択補助として使う。

GUI / CUI イメージ:

```text
Receipt に day_id がある:
  その日の Itinerary を候補として優先表示

Receipt に day_id がない:
  Trip 全体の Itinerary から選択
```

CLI では `--itinerary` 必須を優先し、`day_id` は validation / hint のみに使う（自動選択はしない）。

## 10. `receipt trash` plan

不要な Receipt を Inbox から除外する操作として `receipt trash` を設計する。

CLI 候補:

```bash
caglla receipt trash <receipt_id>
```

方針:

- Receipt を物理削除しない
- `trashed_at` を設定する
- 通常の `receipt list` / pending sum から除外する
- `receipt list --trashed` で表示できるようにする案を検討
- v3.6.0 の `receipt ignore` は deprecated または alias とする案を検討

---

## 11. `receipt restore` / `receipt purge` plan

Trash を導入する場合の操作候補:

```bash
caglla receipt restore <receipt_id>
caglla receipt purge <receipt_id>
```

意味:

| command | Meaning |
|---|---|
| `receipt restore` | Trash から Inbox に戻す |
| `receipt purge` | Trash から完全削除する |

実装スコープ案:

- v3.7.0 初期実装に含める: `restore`（trash が誤操作になり得るため）
- `purge` は Maybe（defer も許容。UI/UX と運用に合わせて判断）

## 12. Pending Receipt sum

目的:

```text
まだ Expense 化されていない支払い候補がいくら残っているかを確認する
```

対象:

```text
active Receipt only（trashed は除外）
```

表示候補:

```text
Pending Receipts:
  count
  total by currency
  without amount count
```

重要:

```text
Pending Receipt sum は Actual ではない。
Planned vs Actual には含めない。
```

概念整理:

```text
Planned = Estimate
Actual  = Expense
Pending = active Receipt Inbox total
```

Potential Actual 表示は future work とする。

## 13. CLI output plan

検討項目:

- `receipt list` に summary を出すか
- `receipt list --json` に summary を含めるか
- `receipt summary` コマンドを別に作るか
- human output と JSON output の差
- 通貨別 total の表示形式
- amount なし Receipt の件数表示
- trashed Receipt を既定で表示しない

最小案（推奨）:

```text
receipt list:
  header summary + list rows

receipt list --json:
  { summary, receipts } envelope
```

JSON 形式は既存の read 系 `--json` と整合するように、互換性（ツール利用）を優先して設計する。

## 14. Export / import schema impact

`trashed_at` や status 廃止を入れる場合、export schema の更新が必要になる可能性がある。

候補:

```text
schema_version: 8
trip.receipts[]
  day_ref optional
  amount optional
  currency optional
  occurred_date optional
  memo optional
  trashed_at optional
```

検討事項:

- v7 import は互換維持
- v6 import も互換維持
- v7 の `status: ignored` は v8 import 時に `trashed_at` へ変換する案
- diff / validate-export の更新が必要
- export で Trash された Receipt を含めるか

推奨（安全側）:

```text
trashed Receipt も export に含める（restore 可能性を保つ）
```

ただし、active-only export の利点（データを「現在の作業対象」に限定）もあるため、最終判断は v3.7.0 実装前の設計確定で行う。

## 16. Planned vs Actual relationship

従来どおり:

```text
Estimate -> Planned
Expense  -> Actual
Receipt  -> Pending（Actual ではない）
```

変更しないもの:

```text
trip export-md Planned / Actual / Difference
trip stats
trip stats --json
Day-level Planned vs Actual
Itinerary-level difference calculation
```

ただし assign 後は Expense が作成されるため、Expense として Actual に反映される。

テスト観点:

```text
Receipt のまま: Planned vs Actual / stats に影響しない
assign 後: Expense として Actual に反映される
```
